# 知食角小程序开发规则

## 0. 最高开发宪法（安全、跨端与边界）

本项目已进入**正式上线运行期**，AI 在辅助开发时必须死守以下三条底线及分工：

1. **极致安全感（环境隔离与数据变更 SOP）**
   - **底线**：绝对禁止直接在 Prod 正式库开发调试！
   - **分工**：用户负责拍板确认“数据可以动”并在控制台做物理操作；AI 只负责写“幂等、可重复执行、有防错保护”的代码，绝不私自执行 `db.remove()`。
   - **数据变更强制 SOP（阶段 3 护栏）**：
     凡是涉及「改字段结构、改用户归属、动核心历史表、批量删除」的高风险需求，必须严格遵循：
     `下达指令 -> AI 在 Test 库写迁移脚本 -> 用户在 Test 库跑通验收 -> 用户手动备份 Prod 库 -> 用户在 Prod 库执行脚本`。
     任何一步缺失，都禁止进行真实数据变更！如果由于极特殊情况必须跳过此 SOP 直接在 Prod 库执行高风险改动，AI 必须强制要求用户（小白）进行**手动三次对话确认**（例如 AI 连续警告三次“你确定要跳过测试直接改正式数据吗？”，用户连续回复三次“确认”），否则 AI 拒绝提供或执行相关代码！
2. **极度防踩坑（跨端友好）**
   - **底线**：任何架构与配置方案（如多环境隔离）必须为未来跨端（iOS/Android，如 uni-app/RN）留有余地。采用“单代码库+本地配置项”的业界标准，禁止使用“多物理文件夹”等不可扩展的死胡同设计。
3. **极度直白可控（拒绝黑盒）**
   - **底线**：拒绝黑盒魔法。改了哪行代码、微信是怎么物理化上传的、环境是怎么切换的，AI 必须用大白话和图解明明白白地解释清楚。
   - **分工**：用户主导方向与业务验收；AI 负责梳理运行原理、提前列出操作后果，并提供自动化执行脚本（如 PowerShell 大扫除脚本）。

> ⚠️ 警告：只有在满足以上三条底线的前提下，才允许进行功能代码的编写与修改。

## 1. PRD 与计划为准

- 每次执行 task 严格按照既定计划（如 `post-launch-safe-iteration-plan.md` 或 `v2-auth-db-plan.md`）推进。
- 有新想法、方案或重大架构改动，必须先出面向小白的“白话图解计划”，与用户达成一致后再执行。
- 不允许擅自偏离 PRD 范围。

## 2. 小步快跑

- 每次只做计划中的一项，或一项中的一小部分。
- 步步优化，处处闭环，确保开发完整性。
- 每个阶段完成后对照 PRD 验收用例自测。
- **强制通道**：任何待上线功能，必须先通过 Test 库体验版的强制回归验证，方可转 Prod 发布。

## 3. 设计参照

- 涉及前端设计参照 `design/` 中的设计图（花钱买的，必须用好）。
- 确保生成的前端效果与设计严格一致。

## 4. 产物归档规范（禁止根目录散落）

- **底线**：绝对禁止在项目根目录（`/`）随意生成临时的 `*.md` 计划文档、JSON 数据样本、或者测试脚本。
- **归档路径**：
  - AI 生成的迭代计划、分析报告等文档必须统一存放在 `docs/` 目录下（如 `docs/post-launch-safe-iteration-plan.md`）。
  - AI 生成的临时测试脚本必须放在 `scripts/` 或 `.trae/scripts/` 中。
- **目的**：保持根目录绝对干净，同时让文档有专属目录统一管理，避免将内部协作文档误打包进小程序发布代码中。

## 5. 后上线时代的演进法则（敬畏现有逻辑）

项目进入优化提升期后，绝不允许“想一出是一出”的破坏性修改。所有代码演进必须遵循以下底线：

- **先溯源，后改写（Context First）**：动手前必须全局检索相关变量、组件、API 的调用链路。没有理清数据流向、生命周期和最初设计意图，禁止直接改写代码。
- **输入输出守恒（I/O Consistency）**：在重构或优化代码（尤其是 Prompt 或云函数逻辑）时，必须保留原逻辑中已有的所有输入维度（如：用户国籍、定位信息等）和输出结构。严禁在未经确认的情况下，擅自精简或丢弃原有的业务参数。
- **最小侵入原则（Minimal Intrusion）**：优先采用“增量扩展”（如新增可选参数、新增独立方法），克制“推翻重写”的冲动。除非明确计划，否则禁止因为一个小需求重构整个模块。
- **老数据与缓存兼容（Backward Compatibility）**：永远假设线上用户带着旧版 Storage 缓存或旧结构的数据。新增字段必须有默认值兜底，修改逻辑必须兼容旧状态，严禁引发老用户白屏或死锁。
- **爆炸半径确认（Blast Radius）**：修改公共组件、全局状态（如 `app.js`）、或云函数核心逻辑前，必须列出所有调用方，并确保演进方案对所有调用方无害。

> **🤖 AI 执行约束**：
> 在处理任何逻辑修改需求时，AI 必须先向用户输出**【逻辑溯源与影响面评估】**（说明原逻辑的 Inputs/Outputs 是什么，改动后会变成什么）。只有在解释清楚“现在的逻辑是怎么运作的”、“改动会影响哪些范围”、“如何兼容老数据”，并获得用户同意后，才能给出具体的代码实现。严禁不看上下文直接强塞代码。
