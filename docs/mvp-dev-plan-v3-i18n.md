# 《真菜》微信小程序 MVP V3 开发计划（国际化双语版本）

在完成了 V2 的个人中心与历史记录后，V3 阶段的核心目标是**面向国际市场，推出“英文优先（默认英文），中文为辅”的双语版本**。这不仅能让全球用户顺畅使用我们的食材识别与菜谱推荐功能，也为未来的海外推广打下坚实基础。

对应 PRD：`docs/mvp-prd.md` 第 9 节
详细方案：`docs/i18n-plan.md`

***

## 0. 你将做出什么（V3 目标）

我们将把小程序从纯中文工具升级为“自适应双语”应用。你最终会拥有：

1. **智能默认语言**：根据用户的微信客户端语言，自动切换为中文（`zh_CN/TW/HK`）或英文（其他所有语言）。
2. **全局语言切换能力**：在「我的」页面提供 `🌐 语言 / Language` 极简入口，点击后底部弹出原生菜单选择语言，全站文字瞬间刷新，无需重启。
3. **双语 AI 识别**：大模型能根据当前语言，输出纯正的英文或中文食材名、鲜度分析和菜谱。
4. **历史数据隔离**：已生成或收藏的历史采购单，按生成时的语言原样展示，不进行强制翻译，保证数据稳定性与低成本。

核心闭环（V3 必须跑通）：

- 英文微信客户端打开小程序 → 界面默认全英 → 拍照识别 → AI 返回全英结果 → 生成全英采购单。
- 在「我的」页面点击语言设置切换为中文 → 界面瞬间变为中文 → 再次识别 → 返回中文结果。

## M13：i18n 基础建设与状态管理 (Infrastructure)

### 目标

建立全局双语字典，并实现“非中文即英文”的自动语言探测机制。

### 你要做什么

1. 在 `miniprogram/utils/` 下新建两个字典文件：
   - `en.js`（英文主干字典）
   - `zh.js`（中文辅助字典）
2. 提取全局的硬编码文本（如“拍照识别”、“正在分析...”等），以 Key-Value 形式存入字典。
3. 在 `app.js` 的 `onLaunch` 中调用 `wx.getSystemInfoSync().language` 探测微信语言：
   - 命中 `zh_CN`、`zh_TW`、`zh_HK` 设为 `zh`。
   - 其他均设为 `en`。
4. 在 `app.js` 中封装一个全局翻译函数 `t(key)` 和一个切换语言的函数 `switchLanguage(lang)`。
5. 优先读取 `wx.getStorageSync('pf_lang')` 作为用户手动选择的语言。

### 验收

- 启动小程序时，能正确在控制台打印出当前的默认语言（en 或 zh）。
- 字典文件结构清晰，能通过 `app.t('camera_title')` 成功获取到对应的文本。

## M14：前端界面双语化改造 (UI Localization)

### 目标

把现有所有页面的中文硬编码替换为动态读取字典的变量，并解决英文排版过长导致的问题。

### 你要做什么

1. 遍历 `pages/index`、`pages/result`、`pages/list`、`pages/my`：
   - 将 WXML 中的中文替换为 `{{i18n.xxx}}` 变量。
   - 将 JS 文件中的 `wx.showToast` 等提示语替换为 `t('xxx')`。
   - 分享文案、弹窗文案等也全部替换。
2. 在所有页面的 `onLoad` 和 `onShow`（或自定义的全局监听器）中，将 `app.globalData.i18n` 绑定到页面的 `data.i18n`。
3. **排版防崩溃（关键）**：
   - 在 `app.wxss` 中全局增加 `word-break: break-word`。
   - 检查卡片标题、按钮、长文本区域，确保英文文本即使很长也会自动换行，不会撑破容器。

### 验收

- 手动在 `app.js` 中把语言写死为 `en`，编译后四个页面的所有静态文字都变成了英文。
- 英文界面下的按钮和文本没有溢出屏幕或遮挡其他元素。

## M15：云函数大模型双语适配 (Backend AI Adaptation)

### 目标

让云端大模型知道当前用户用的是什么语言，并输出对应语言的结构化 JSON。

### 你要做什么

1. 在小程序端 `utils/analyze.js` 调用云函数时，额外传入 `lang` 参数（`app.globalData.language`）。
2. 修改 `cloudfunctions/analyze/index.js`：
   - 接收 `lang` 参数。
   - 根据 `lang` 动态拼接 Prompt。如果是 `en`，要求大模型以英文输出 `ingredient_name`、`taste`、`texture`、`freshness_reason`、`recipes` 等，且不要夹杂中文。
3. 保证云函数返回的 JSON 结构中英文字段完全一致。
4. 补充/分离 Mock 数据，根据语言返回 `mock-en` 或 `mock-zh`。

### 验收

- 开启云开发，英文环境下拍照识别，结果页展示的食材名、鲜度分析、做法全都是英文。
- JSON 结构没有因为语言改变而遭到破坏。

## M16：交互体验闭环 (Interaction & UX)

### 目标

在「我的」页面增加语言切换入口，实现无刷新秒切体验，并处理历史记录的隔离展示。

### 你要做什么

1. 在 `pages/my/index.wxml` 增加一行菜单：`🌐 语言 / Language`。
2. 点击菜单弹出原生 `wx.showActionSheet`，提供“跟随系统（默认）”、“简体中文”、“English”等选项。
3. 用户选择后触发 `switchLanguage`，将新语言存入 `Storage` 并更新 `globalData`，触发当前页面的 `setData({ i18n: newI18n })`，让文字瞬间变化。
4. **历史记录隔离**：确保 `pages/my` 列表和 `pages/list` 采购单渲染的历史数据（从数据库拉取的 `ingredient_name` 等）直接使用原始数据，不走翻译字典（UI 框架文字走翻译，内容走原样）。

### 验收

- 在「我的」页面点击语言设置，弹出底部菜单，选择新语言后页面文字瞬间变化。
- 进入“我的”页面，之前用中文识别存下的历史记录依然显示中文，但页面的空状态提示等 UI 元素是英文。

## M17：全链路测试与边界排查 (QA & Testing)

### 目标

完成国际化改造的最后一公里，确保双语版本没有遗漏的死角。

### 你要做什么

1. **默认语言测试**：在微信开发者工具中模拟更改客户端语言（英文、简体中文），查看启动默认语言是否正确。
2. **长文本 UI 跑查**：刻意找一些英文名特别长的食材（如 "Mediterranean Sea Bass"），查看结果页和采购单页的换行效果。
3. **AI 纯净度检查**：检查英文 AI 结果是否有“中式英语”或偶尔夹杂的汉字，如有，继续调优云函数 Prompt。
4. **细节排查**：检查所有 `wx.showToast`、`wx.showModal` 以及用户分享给朋友的小程序卡片标题，确保都正确应用了多语言。

### 验收

- 全部 5 个核心页面（首、结果、列表、我的、分享态）在双语下表现完美。
- AI 识别在中英双语下都保持了高可用性。

***

## M18：翻译与排版精细打磨 (Translation & UI Polish)

### 目标

在核心双语功能完全调通后，专门针对英文翻译的准确性、地道程度，以及英文较长导致的排版细节进行精细化打磨，中英文风格一致。

### 你要做什么

1. **翻译校对**：全面走查 `en.js` 中的词汇，确保符合海外用户的表达习惯，消除“机翻感”。
2. **排版调优**：针对英文长文本，微调字号、行高、间距 (padding/margin)，确保卡片、按钮在不同机型下都能保持设计图的美观度。
3. **极值处理**：对极个别确实过长的文本，增加合理的折行、省略号 (`text-overflow: ellipsis`) 或自适应缩小字号机制。

### 验收

- 中英双语切换时，界面不仅布局稳定，而且在两种语言下都显得自然、专业、美观。
- 翻译用词准确地道。
