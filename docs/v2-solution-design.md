# 识为鲜 V2 版本解决方案设计文档 (V2 Solution Design)

> 本文档用于记录针对 `v2-issues-tracking.md` 中各类问题的具体产品与技术解决方案，作为 V2 开发的架构指导。

## 二、核心交互与UI重构（针对第二类问题）

### 1. 首页UI平衡与重构（解决 [问题 2]）
**产品思考**：V1 只有“识物”，V2 加了“记一笔清单”。如果首页把这两个入口做得一样大，用户会失去焦点，产品定位也会变得模糊。
**UI/UX 方案：渐进式增强的主次分明设计**
* **视觉重心（主线）**：保持“📷 扫一扫/相册上传”为页面的绝对核心（C位大按钮），延续 V1 “探索世界”的核心 Aha Moment。
* **快捷入口（辅线）**：将“🎤 记一笔”作为悬浮的 FAB (Floating Action Button) 或放置在页面中下部一个稍小但极易触达的胶囊按钮。
* **情境信息卡片**：如果用户当前有未完成的清单，首页上半部分展示一个精美的“半折叠卡片”，提示“你有 3 个食材待购买”，点击可直达清单页。
**验证方法**：使用开发者工具预览首页，验证主入口和辅入口的点击响应，检查未完成清单卡片在有/无数据情况下的条件渲染是否正确。

### 2. 输入交互扁平化（解决 [问题 3]）
**产品思考**：用户在厨房或者路上想到要买什么，掏出手机记下来是一个“高频且追求极速”的动作。原来的“点击记一笔 -> 选择文本还是语音 -> 再输入”层级太深了。
**UI/UX 方案：左右滑动切换的底部抽屉**
* 取消中间跳转页。点击“记一笔”后，直接从底部弹出一个 Half-Screen 抽屉面板。
* **交互设计**：
  * 面板默认停留在**左侧 [文本输入态]**：自动唤起系统软键盘，方便用户快速打字。
  * 用户**向右滑动**面板（或点击 Tab）：无缝切换到**右侧 [语音输入态]**，展示一个动感的麦克风按钮，长按或点击即可说话。
* **体验亮点**：利用 `swiper` 或原生手势监听，让输入方式的切换就像翻页一样顺滑。
**验证方法**：在真机上验证点击按钮后抽屉弹出的流畅度；测试手指左右滑动能否正常切换文本域和语音录制组件；验证文本态是否能正确自动拉起键盘。

### 3. 逐道菜选购排版优化（解决 [问题 7]）
**技术方案**：
* 修复“搜索做法”区域与食材列表区域的层级或高度重叠问题。
* 检查 CSS 的 `z-index` 或 `flex` 布局高度分配。针对不同机型（尤其是有刘海或底部安全区的全面屏）使用 `safe-area-inset-bottom` 进行适配。
**验证方法**：在微信开发者工具中切换 3-4 种不同屏幕尺寸（如 iPhone SE, iPhone 14 Pro Max 等），检查规划页面是否出现遮挡或滚动截断现象。

### 4. Icon 替换（解决 [问题 4]）
**UI/UX 方案**：
* 目前清单页的 Icon 如果和首页重复，会引发用户认知混乱。
* 建议将清单页的底部 Tab Bar Icon 替换为清晰的“购物车（Cart）”或“带打勾的便签本（List）”图标。
**验证方法**：检查 `app.json` 中 `tabBar` 配置的 `iconPath` 和 `selectedIconPath`，验证真机显示效果是否清晰区分。

## 一、核心业务逻辑与底层数据架构（针对第一类问题）

### 1. 核心模块关系梳理（解决 [问题 10]）

**架构方案：Single Source of Truth（唯一数据源）**

- **底层数据源**：采用两张核心表管理数据，避免数据冗余和不一致。
  - `ShoppingList` 表：代表\*\*“每次购买”\*\*（例如：2026-04-03 采购单），包含 `status`（`active`进行中 / `completed`已完成）。
  - `Ingredient` 表：代表**“每个食材”**，挂载在某个 `list_id` 下。携带关键维度字段：
    - `source_recipe`（所属菜谱）
    - `category`（超市动线分类）
    - `is_main_ingredient`（是否为主食材，由拍照识物产生的主体）
    - `source_image_id`（关联的拍照/上传图片 ID，用于集邮功能和UI溯源）
- **UI 层动态映射（不存冗余数据）**：
  - **逐道菜选购 (Planning View)**：前端对当前 `active` 清单下的 `Ingredient`，按 `source_recipe` 进行 `GroupBy`（分组）渲染。如果分组内包含 `is_main_ingredient = true` 的食材，食谱标题处需打上特殊标记（如：带缩略图或高亮标识）。
  - **汇总选购 (Execution View)**：前端对当前 `active` 清单下的 `Ingredient`，按 `standard_name` 进行数量累加合并，再按 `category` 进行 `GroupBy` 渲染。

**验证方法 (Verification)**：
1. **Mock 数据验证**：在数据库手动插入包含主食材和普通辅料的 mock 数据，验证前端是否能正确根据 `source_recipe` 分组，且食谱标题正确带出照片缩略图/主食材标识。
2. **状态机测试**：修改一个 `Ingredient` 的 `status`，验证逐道菜选购和汇总选购的列表是否同步响应更新，证明“唯一数据源”生效。

### 2. 业务闭环与 CRUD 管理（解决 [问题 8]）

**架构方案：基于视角的颗粒度降维打击**

- **业务闭环点**：用户在“汇总选购”点击【完成采购】。触发闭环，当前 `ShoppingList` 状态变为 `completed`。未勾选的食材，系统弹窗询问：“有部分食材未购买，是否移入下一次采购单？”。如果勾选了主食材（`is_main_ingredient=true`），则该主食材将点亮用户的“食材集邮册”。
- **CRUD 权限分配**：
  - **按“每道菜”（逐道菜选购操作）**：允许用户左滑整道菜（如：糖醋排骨）点击“删除”。底层逻辑：将该 `list_id` 下所有 `source_recipe = 糖醋排骨` 的食材状态标为 `deleted`。
  - **按“每个食材”**：
    - **逐道菜选购下**：允许直接修改单个食材的数量、单位及删除。
    - **汇总选购下**：因为显示的是聚合数据，**只允许划线打勾（改变状态为** **`bought`），不允许修改数量**。如需修改，引导用户切回“逐道菜选购”操作，极大降低代码复杂度和用户的认知负担。
  - **按“每次购买”**：只能在“历史记录”中进行整单删除或“一键复购”。

**验证方法 (Verification)**：
1. **边界权限验证**：模拟用户在汇总选购点击数量编辑框，验证是否拦截操作并引导至逐道菜选购。
2. **批量状态流转验证**：模拟在逐道菜选购左滑删除整道菜，查库验证所有从属于该 `source_recipe` 的关联 `Ingredient` 记录的 `status` 是否都更新为 `deleted`。

### 3. 历史记录视角拆分（解决 [问题 5]）

**架构方案：将历史模块分为三 Tab**

- **Tab 1：按每次购买（执行历史/账单视角）**
  - **呈现形式**：时间轴瀑布流（如：2026-04-03 采购单、2026-04-01 采购单...）。
  - **用户价值**：点击进入可查看当时的采购明细，提供【一键复购】按钮，将这单的常备菜（如葱姜蒜）原封不动加入今日待买清单。
- **Tab 2：按每道菜（私人菜谱库/逐道菜选购）**
  - **呈现形式**：聚合用户过去所有 `completed` 清单中的 `source_recipe`（去重），形成菜品卡片（如：红烧肉、蒜蓉西蓝花）。
  - **用户价值**：解决“今天不知道吃什么”的痛点。用户浏览历史菜谱，点击【加入今日清单】，系统自动把当年做该菜的所需食材提取到今天的 `active` 采购单中。
- **Tab 3：按主食材（集邮/探索视角）**
  - **呈现形式**：精美的“食材图鉴”网格瀑布流。按食材种类（海鲜、菌菇）分类展示。已购买（或已识物）的 `is_main_ingredient=true` 的食材高亮点亮，并显示关联的 `source_image_id` 缩略图；未购买的显示剪影。
  - **用户价值**：满足猎奇、收集癖与社交分享欲，也是 V1 核心识物价值的沉淀。

**验证方法 (Verification)**：
1. **聚合查询验证**：编写测试脚本，验证从 `Ingredient` 表分别通过按 `list_id` 分组、按 `source_recipe` 去重、按 `is_main_ingredient=true` 筛选，能够准确渲染出三个 Tab 的 mock 视图。
2. **复购功能链路验证**：在 Tab 1 或 Tab 2 点击复购，验证当前 `active` 的 `ShoppingList` 下是否正确新增了相应的 `Ingredient` 记录（包含正确的 `source_recipe` 继承）。
