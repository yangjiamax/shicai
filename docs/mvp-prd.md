# 《知食公社》微信小程序 MVP PRD（可快速 Demo）

## 1. 目标

- 跑通闭环：拍照/上传 → 识别结果 → 鲜度判断 → 做法推荐 → 佐料采购单
- 用最小成本验证：用户是否愿意在菜市场使用「拍一下就给出做法与佐料」

## 2. MVP 范围（P0）

### 2.1 页面

- 首页（拍照/上传）
- 结果页（食材名称 + 鲜度结论 + 推荐做法）
- 采购单页（佐料清单勾选 + 一键分享）

### 2.2 功能

- 拍照/相册上传图片
- 启动时建立登录态（静默）：获取/生成 userId（openid 或匿名 id）并本地持久化，用于基础统计、限流与后续扩展
- 单次分析返回结构化结果（见接口约定）
- 结果页选择一个做法并生成采购单
- 采购单勾选与微信原生分享小程序卡片
- 采购单收藏功能：允许用户收藏佐料清单到云端，可在"我的"页面查看
- 提供「退出登录/清除本地数据」入口：清空本地缓存并重新建立会话（不涉及昵称头像/手机号等完整账号体系）

## 3. 暂不做（后续迭代）

- 识别准确率/耗时的硬指标承诺与优化工程
- 英文名输出、地区特色菜匹配、成品图版权与检索
- 鲜度形态识别（整条/切块/剥壳）与置信度百分比
- 采购单按摊位顺序分组、海报/小程序码定制生成
- 完整账号合并、运营后台

## 4. 用户主路径

1. 打开小程序 → 静默登录（建立会话/拿到 userId）→ 进入首页
2. 点击拍照/上传
3. 进入加载态（文案可趣味化）→ 获取结构化分析结果
4. 展示：食材名/简介、鲜度档位与依据、2-3 个做法
5. 选择做法 → 生成佐料采购单
6. 采购单勾选 → 分享给家人/朋友 → 再拍一次

## 5. 关键数据结构（概念）

分析结果对象：

- ingredient_name：食材中文名
- ingredient_desc：纯客观的一句话简介
- taste：口味（如鲜甜、浓郁）
- texture：口感（如紧实、细嫩）
- similar：类似食材（作为认知锚点）
- freshness_level：新鲜｜一般｜不太新鲜
- freshness_reason：1 句依据（将腐败/死亡等细节限制在此处）
- recipes：做法列表对象（2026-04-01 跨文化体验升级）
  - familiar：熟悉的滋味做法数组（2-3 个）
    - recipe_name：做法名
    - ingredients_needed：佐料数组（字符串）
  - local：当地的做法数组（2-3 个）
    - recipe_name：做法名
    - ingredients_needed：佐料数组（字符串）

## 5.1 登录与退出（P0 口径）

目标：不做复杂账号体系，但必须有稳定的「用户会话标识」与「清除本地数据」能力，保证后续可扩展（历史/收藏/配额等）。

- 登录态定义：小程序启动后获得 `userId`，优先使用云端可提供的 openid；无法获取时降级为匿名 `userId`（本地生成并持久化）
- 登录方式：静默，不弹出昵称头像授权弹窗；不依赖 `getUserProfile`
- 退出定义：清空本地缓存（含 `userId`/token 与业务缓存），回到“未登录态”，随后在下一次启动/下一次需要请求时重新建立会话

建议本地存储键（实现口径）：

- `pf_user_id`: string
- `pf_auth_source`: "cloud\_openid" | "wx\_login" | "anonymous"
- `pf_cache_*`: 业务缓存（如有）

## 6. 验收标准（MVP）

- 不开启云开发也能跑通页面闭环（Mock 数据）
- 开启云开发后可通过云函数返回结构化结果并正常渲染
- 冷启动时可自动建立登录态并获得 `userId`（云开发优先 openid；无云开发则匿名 id）
- 执行「退出登录/清除本地数据」后，本地缓存被清空且不影响继续使用核心闭环（可重新建立会话）
- 选择做法后可生成采购单并勾选
- 分享按钮可正常转发小程序卡片

## 8. MVP V2 迭代说明（个人中心与历史记录）

在 V1 跑通基础识别与生成采购单后，我们需要通过“个人中心”和“历史记录”来增强用户的粘性与多食材回溯能力，并为未来的商业化（付费限额）做铺垫。

### 8.1 V2 核心功能范围

1. **多食材识别与回溯（历史记录）**：
   - 首页增加“我的历史”入口，或独立的“我的”页面。
   - 记录用户识别过的食材、选择的做法及生成的佐料清单。
   - 数据存储方案：基于 `userId`（openid），将历史记录持久化至云开发数据库（Cloud DB），保证跨设备一致性。

2. **用户体系与显式登录（个人中心）**：
   - 新增「我的」页面（User Center）。
   - 提供“点击登录/设置头像昵称”的功能（使用微信官方 `chooseAvatar` 和 `nickname` 填写能力）。
   - 将用户的微信头像和昵称与现有的 `userId` 绑定并保存至云数据库。
   - 目的是为用户提供专属感（个人展示）、历史同步，以及为后续的分享裂变和商业化能力建立账号基础。

### 8.2 V2 页面流转补充

```text
[我的/个人中心页]（新页面）
  ├─ 顶部：头像 + 昵称（未登录时显示默认头像和“点击登录”）
  │   └─ 点击 -> 弹窗/页面：获取微信头像和昵称 -> 更新云端用户信息
  ├─ 核心功能：
  │   ├─ 国籍 / Nationality 设置 (2026-04-01 新增，跨文化体验依赖)
  │   ├─ 识别历史：展示过往识别记录列表（按时间倒序）
      │   └─ 点击某条记录 -> 统一进入[结果页]（还原完整上下文，保持动线复用）
      └─ 其他：清除本地缓存、关于我们、语言切换等
```

### 8.3 V2 数据库结构设计（概念）

**User 集合 (`users`)**：
- `_id` / `userId`: string (用户的 OpenID 或系统分配 ID)
- `avatarUrl`: string (头像路径，可能需要存至云存储)
- `nickName`: string (用户填写的昵称)
- `nationality`: string (用户设置的国籍，2026-04-01 新增)
- `createdAt`: Date
- `updatedAt`: Date

**History 集合 (`histories`)**：
- `_id`: string (记录唯一 ID)
- `userId`: string (关联用户)
- `imageUrl`: string (原图/缩略图云存储 ID)
- `analysisResult`: object (完整的识别结果对象，含食材名、简介、鲜度、所有做法等上下文)
- `selectedRecipe`: object (用户选中的做法和最终勾选的佐料清单)
- `createdAt`: Date

**Favorites 集合 (`favorites`)**：
- `_id`: string (记录唯一 ID)
- `userId`: string (关联用户)
- `ingredient_name`: string (食材名)
- `recipe_name`: string (做法名)
- `ingredients_needed`: array (佐料数组，用户勾选后的结果)
- `createdAt`: Date

## 9. 国际化 (i18n) 设计（2025-03-31 新增）

> **目的**：面向国际市场，推出"英文优先（默认英文），中文为辅"的双语版本小程序，让全球用户都能便捷使用知食公社的食材识别、鲜度判断和菜谱推荐功能。

### 9.1 核心产品决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 切换入口位置 | 「我的」页面 | 遵循“无感即最佳体验”，将低频的语言切换收纳至“我的”页面。提供 `🌐 语言 / Language` 入口，点击后弹出原生 ActionSheet 供用户选择（跟随系统/中文/英文） |
| 默认语言策略 | 跟随微信系统，非中文即英文 | 自动读取微信客户端语言，如果是 `zh_CN`、`zh_TW`、`zh_HK` 则默认中文；否则默认英文 |
| 历史数据处理 | 保持原有语言 | 已生成/收藏的采购单和识别结果保持生成时的语言，不实时翻译，降低成本并保证稳定性 |

### 9.3 共用 vs 分开的划分原则

#### ✅ 应该共用的部分（Single Source of Truth）

| 类别 | 共用原因 | 示例 |
|------|----------|------|
| 架构基础设施 | 与语言无关 | 云函数调用封装 `utils/analyze.js`、认证模块 `utils/auth.js`、小程序初始化配置 |
| 业务逻辑 | 跨语言完全一致 | 页面跳转逻辑、图片选择逻辑、存储读写逻辑、数据解析逻辑 |
| UI 结构与样式 | 布局一致，只是文本长度不同 | 所有 `.wxss` 中的布局、Flex 盒子、组件结构 |
| 非文本资源 | 语义化图标全球通用 | `images/icons/` 下所有 SVG 图标、背景图 `market-bg.jpg` |

#### ⚡️ 应该分开的部分（Localized）

| 类别 | 分开原因 | 示例 |
|------|----------|------|
| 硬编码文本 | 必须翻译 | 导航栏标题 "知食公社"、按钮文案 "拍照识别"、提示语 "正在分析..."、错误消息 |
| 动态翻译内容 | 由 AI 根据语言返回不同版本 | 食材名称、菜谱名、鲜度描述、佐料列表（这部分是"数据层"的分层） |

### 9.4 全局语言状态管理

- **冷启动探针**：小程序初始化时调用 `wx.getSystemInfoSync().language` 探测微信客户端语言
- **默认规则**：`zh_CN` / `zh_TW` / `zh_HK` → 中文；其他所有语言（包括 `en`, `ja`, `ko` 等）→ 英文
- **用户覆盖**：将用户手动切换的结果存入 `wx.setStorageSync('pf_lang')`，优先级高于系统默认
- **全局函数**：在 `app.js` 中统一定义 `t(key)` 翻译函数，供所有页面调用

### 9.5 AI 云端双语适配

- **语言参数传递**：小程序端调用 `analyze` 云函数时，传入当前语言标识 `{ lang: 'en' | 'zh' }`
- **动态 Prompt**：云函数根据 `lang` 参数动态生成中/英文 Prompt，确保大模型输出对应语言
- **结构一致性**：无论中英文模式，AI 返回的 JSON 结构完全一致（字段名相同，只是值用不同语言），前端解析代码无需改动
- **默认兜底值**：云函数的默认值（如 `"简单清炒"`）也需要根据语言返回对应版本

### 9.6 兼容性设计策略

1. **新鲜度映射表**：在 `app.js` 中维护中英文映射表
   ```javascript
   const freshnessMap = {
     zh: { '新鲜': 'fresh', '一般': 'normal', '不太新鲜': 'bad' },
     en: { 'Fresh': 'fresh', 'Average': 'normal', 'Not Fresh': 'bad' }
   };
   ```

2. **佐料连接符**：统一使用 `i18n.list_separator`（中文 `"、"`，英文 `", "`）

3. **分享文案模板**：
   - 中文：`"今晚准备做【${name}】，快来看看需要买什么！"`
   - 英文：`"Tonight we're making 【${name}】! Here's what you need to buy:"`

4. **Mock 数据分离**：根据 `app.globalData.language` 加载对应的 Mock 数据文件

### 9.7 国际化影响点清单（风险评估）

> 具体影响点详见 `docs/i18n-plan.md`，以下是核心变更汇总：

| 风险等级 | 影响范围 | 数量 |
|----------|----------|------|
| ⚠️ 高风险 | 需要改动的核心文案和逻辑（UI 文本、Prompt、映射表） | 约 40+ 处 |
| 🔶 中风险 | 需要适配的连接符、默认值、Mock 数据 | 约 10 处 |

### 9.8 验收标准（国际化）

1. 切换到英文后，所有静态 UI 文本显示为英文；切换到中文后显示为中文
2. 英文模式下，AI 返回的食材名称、菜谱名、佐料为英文；中文模式下为中文
3. 分享给好友的卡片标题和描述根据当前语言正确生成
4. 历史记录（中文环境生成）切换到英文后，保持原语言不翻译
5. 英文模式下，长文本正确换行，不溢出容器

## 10. 跨文化饮食体验功能设计 (2026-04-01 新增)

> **目的**：在旅游场景下，让产品从“实用工具”升级为“有温度的探索服务”，为用户提供“熟悉的滋味”（基于国籍）和“试试当地做法”（基于地理位置）。

### 10.1 核心机制

1. **国籍阻断与保存**：
   - 用户在首页点击拍照或上传相册时，若未设置国籍，强制跳转至「我的」页面进行设置。
   - 国籍信息一次性输入，优先本地保存（`userNationality`），后续按需同步云端。
2. **静默定位**：
   - 发起识别前静默获取用户地理位置（`latitude`, `longitude`）。
   - 设 3 秒超时保护，拒绝授权或获取失败则传 `null`，不阻断核心流程。
3. **AI Prompt 与结构升级**：
   - 云函数入参增加 `nationality` 和 `location`。
   - AI 根据 `nationality` 推荐家常做法（熟悉的滋味），根据 `location` 推荐当地做法（试试当地做法）。
   - 输出结构从数组变更为对象：`{ "familiar": [...], "local": [...] }`。

### 10.2 兼容性与 UI 改造

- **双 Tab 展示**：结果页引入双 Tab 设计（[🍴 熟悉的滋味] / [📍 试试当地做法]）。
- **向下兼容**：若 `local` 数据为空（无定位或AI未生成），或读取旧历史记录（`recipes` 为旧版数组），则隐藏 Tab 栏，平铺展示菜谱，确保页面不报错。

### 10.3 验收标准（跨文化体验）

1. 首次使用未选择国籍，首页拍照被阻断并跳转至“我的”页面。
2. 选择国籍后，重新拍照，正常进入结果页。
3. 授权定位：结果页正常展示“熟悉的滋味”和“试试当地做法”双 Tab。
4. 拒绝授权定位或断网：结果页仅展示“熟悉的滋味”平铺列表，不报错。
5. 切换系统语言（中/英）：新增的提示、Tab 标题、选择器选项能正确翻译。
6. 打开旧的历史记录：完美兼容展示，不出现空白或报错。
