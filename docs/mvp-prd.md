# 《识材》微信小程序 MVP PRD（可快速 Demo）

## 1. 目标

- 跑通闭环：拍照/上传 → 识别结果 → 鲜度判断 → 做法推荐 → 佐料采购单
- 用最小成本验证：用户是否愿意在菜市场使用「拍一下就给出做法与佐料」

## 2. MVP 范围（P0）

### 2.1 页面

- 首页（拍照/上传）
- 结果页（食材名称 + 鲜度结论 + 推荐做法）
- 采购单页（佐料清单勾选 + 一键分享）

### 2.2 功能

- 拍照/相册上传图片
- 启动时建立登录态（静默）：获取/生成 userId（openid 或匿名 id）并本地持久化，用于基础统计、限流与后续扩展
- 单次分析返回结构化结果（见接口约定）
- 结果页选择一个做法并生成采购单
- 采购单勾选与微信原生分享小程序卡片
- 采购单收藏功能：允许用户收藏佐料清单到云端，可在"我的"页面查看
- 提供「退出登录/清除本地数据」入口：清空本地缓存并重新建立会话（不涉及昵称头像/手机号等完整账号体系）

## 3. 暂不做（后续迭代）

- 识别准确率/耗时的硬指标承诺与优化工程
- 英文名输出、地区特色菜匹配、成品图版权与检索
- 鲜度形态识别（整条/切块/剥壳）与置信度百分比
- 采购单按摊位顺序分组、海报/小程序码定制生成
- 完整账号合并、运营后台

## 4. 用户主路径

1. 打开小程序 → 静默登录（建立会话/拿到 userId）→ 进入首页
2. 点击拍照/上传
3. 进入加载态（文案可趣味化）→ 获取结构化分析结果
4. 展示：食材名/简介、鲜度档位与依据、2-3 个做法
5. 选择做法 → 生成佐料采购单
6. 采购单勾选 → 分享给家人/朋友 → 再拍一次

## 5. 关键数据结构（概念）

分析结果对象：

- ingredient_name：食材中文名
- ingredient_desc：纯客观的一句话简介
- taste：口味（如鲜甜、浓郁）
- texture：口感（如紧实、细嫩）
- similar：类似食材（作为认知锚点）
- freshness_level：新鲜｜一般｜不太新鲜
- freshness_reason：1 句依据（将腐败/死亡等细节限制在此处）
- recipes：做法列表（2-3 个）
  - recipe_name：做法名
  - ingredients_needed：佐料数组（字符串）

## 5.1 登录与退出（P0 口径）

目标：不做复杂账号体系，但必须有稳定的「用户会话标识」与「清除本地数据」能力，保证后续可扩展（历史/收藏/配额等）。

- 登录态定义：小程序启动后获得 `userId`，优先使用云端可提供的 openid；无法获取时降级为匿名 `userId`（本地生成并持久化）
- 登录方式：静默，不弹出昵称头像授权弹窗；不依赖 `getUserProfile`
- 退出定义：清空本地缓存（含 `userId`/token 与业务缓存），回到“未登录态”，随后在下一次启动/下一次需要请求时重新建立会话

建议本地存储键（实现口径）：

- `pf_user_id`: string
- `pf_auth_source`: "cloud\_openid" | "wx\_login" | "anonymous"
- `pf_cache_*`: 业务缓存（如有）

## 6. 验收标准（MVP）

- 不开启云开发也能跑通页面闭环（Mock 数据）
- 开启云开发后可通过云函数返回结构化结果并正常渲染
- 冷启动时可自动建立登录态并获得 `userId`（云开发优先 openid；无云开发则匿名 id）
- 执行「退出登录/清除本地数据」后，本地缓存被清空且不影响继续使用核心闭环（可重新建立会话）
- 选择做法后可生成采购单并勾选
- 分享按钮可正常转发小程序卡片

## 8. MVP V2 迭代说明（个人中心与历史记录）

在 V1 跑通基础识别与生成采购单后，我们需要通过“个人中心”和“历史记录”来增强用户的粘性与多食材回溯能力，并为未来的商业化（付费限额）做铺垫。

### 8.1 V2 核心功能范围

1. **多食材识别与回溯（历史记录）**：
   - 首页增加“我的历史”入口，或独立的“我的”页面。
   - 记录用户识别过的食材、选择的做法及生成的佐料清单。
   - 数据存储方案：基于 `userId`（openid），将历史记录持久化至云开发数据库（Cloud DB），保证跨设备一致性。

2. **用户体系与显式登录（个人中心）**：
   - 新增「我的」页面（User Center）。
   - 提供“点击登录/设置头像昵称”的功能（使用微信官方 `chooseAvatar` 和 `nickname` 填写能力）。
   - 将用户的微信头像和昵称与现有的 `userId` 绑定并保存至云数据库。
   - 目的是为用户提供专属感（个人展示）、历史同步，以及为后续的分享裂变和商业化能力建立账号基础。

### 8.2 V2 页面流转补充

```text
[我的/个人中心页]（新页面）
  ├─ 顶部：头像 + 昵称（未登录时显示默认头像和“点击登录”）
  │   └─ 点击 -> 弹窗/页面：获取微信头像和昵称 -> 更新云端用户信息
  ├─ 核心功能：
  │   ├─ 识别历史：展示过往识别记录列表（按时间倒序）
  │   │   └─ 点击某条记录 -> 重新进入[采购单页]或[结果页]
  │   └─ 其他：清除本地缓存、关于我们等
```

### 8.3 V2 数据库结构设计（概念）

**User 集合 (`users`)**：
- `_id` / `userId`: string (用户的 OpenID 或系统分配 ID)
- `avatarUrl`: string (头像路径，可能需要存至云存储)
- `nickName`: string (用户填写的昵称)
- `createdAt`: Date
- `updatedAt`: Date

**History 集合 (`histories`)**：
- `_id`: string (记录唯一 ID)
- `userId`: string (关联用户)
- `imageUrl`: string (原图/缩略图云存储 ID，可选)
- `ingredient_name`: string (食材名)
- `selected_recipe`: object (选中的做法和佐料)
- `createdAt`: Date

**Favorites 集合 (`favorites`)**：
- `_id`: string (记录唯一 ID)
- `userId`: string (关联用户)
- `ingredient_name`: string (食材名)
- `recipe_name`: string (做法名)
- `ingredients_needed`: array (佐料数组，用户勾选后的结果)
- `createdAt`: Date
