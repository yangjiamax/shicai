# 识为鲜多端账号体系演进规划（远期）

本文档保存了关于「识为鲜」从小程序单端走向 iOS/Android 多端互通的远期账号体系架构升级规划。

## 一、 现状评估与痛点分析

当前架构特点：基于微信云开发，依赖 `auth.js` 静默获取 OpenID，无真实登出，强制 Onboarding 收集头像/昵称和权限。

### 1. 产品视角
*   **虚假的“退出登录”**：当前 `logout` 的逻辑仅仅是清空本地缓存，然后立刻重新执行 `initAuth()` 静默获取同一个 OpenID。用户无法实现“切换账号”或“彻底注销”，存在合规和体验风险。
*   **跨端身份断层**：OpenID 是小程序特有的。如果用户未来下载了 iOS App，即使他们用同一个微信号登录，App 获取到的也是另外的 OpenID，导致数据无法互通。
*   **转化率漏斗风险**：强制 Onboarding 收集资料，增加新用户流失率。
*   **匿名用户数据孤岛**：云函数登录失败时生成的匿名数据，在用户真正授权后缺乏合并机制。

### 2. 研发视角
*   **核心标识强耦合**：代码和数据库强依赖 `_openid` 作为主键，这是多端架构的致命伤。
*   **强绑定微信云开发生态**：未来要支持 App 或 Web，必须重构出一套标准的后端服务。
*   **缺乏标准的 Token 鉴权**：目前基于 OpenID 裸奔，缺乏 JWT 或 Session 机制。

## 二、 多端账号体系升级方案（远期 4 阶段）

### 阶段一：数据模型重构
1.  **引入全局 `user_id`**：废弃 `_openid` 作为主键。系统内部分配全局唯一的 UUID。
2.  **建立身份映射表 (`user_identities`)**：一个 `user_id` 绑定多个维度的登录方式（`phone`, `wechat_unionid`, `apple_id` 等）。

### 阶段二：后端鉴权服务建设
1.  **搭建独立后端服务**：使用 Node.js/Go 等，脱离纯云开发环境。
2.  **实现核心 Auth 接口**：提供获取 UnionID、手机号验证、发放 JWT Token 等标准接口。
3.  **匿名数据合并机制**：提供接口合并匿名时期产生的数据。

### 阶段三：小程序端重构
1.  **重写 `auth.js`**：改为调用后端 API，本地存储 token 并携带在请求头中。
2.  **改造 Onboarding 与登录逻辑**：允许游客态，按需触发登录与授权。
3.  **实现真正的 Logout**：清空 token 并跳转至真实 Login 页面。

### 阶段四：App 端扩展准备
1.  **接入原生 SDK**：App 端集成微信 SDK 获取 UnionID 实现账号互通。
2.  **Apple 登录**：iOS App 强制合规要求。
3.  **手机号一键登录**：作为多端通用的基准账号。
