# 微信小程序 MiniTest 性能检测报告问题清单

> **报告时间**：2026-04-16
> **测试类型**：智能化 Monkey 测试
> **综合运行性能得分**：82 分（小米 11 Ultra: 85分，iPhone 11: 80分）
> **异常检测**：JSError、网络请求、UI检测均全部通过（0 异常）。

以下是根据 MiniTest 测试报告中“待改进”项如实汇总的性能问题清单。为了后续优化排期，已将这些问题分为四大类：

## 一、 启动与首屏性能问题（包袱过重）
影响小程序的冷启动速度和首屏展示时间：
1. **页面 App.onLaunch 前存在耗时操作**（如 `app.js` 中的全局代码同步执行过多）。
2. **存在 App 生命周期执行耗时过长的情况**。
3. **启动阶段存在重复获取相同 key 的 storage 信息**。
4. **首屏渲染前多次调用同步 API**。

## 二、 数据通信与逻辑层问题（带宽浪费）
影响页面交互流畅度和跳转速度：
5. **setData 传入未绑定在 WXML 的变量**（即把不需要在视图层渲染的冗余数据也一并 setData 了，浪费通信带宽）。
6. **页面渲染前多次调用同步 API**。
7. **重复获取相同 key 的 storage 信息**（不仅在启动阶段，在普通页面运行期间也存在此现象）。
8. **页面跳转响应慢**（在低端机型如 iPhone 11 上尤为明显，通常是由目标页面 onLoad/onShow 中同步任务过多或 setData 数据过大导致）。

## 三、 视图层渲染压力问题
影响页面的滑动体验，容易造成卡顿：
9. **过大的 WXML 节点数目**（大概率出现在历史记录、清单列表或食谱详情等存在长列表的页面，缺乏虚拟列表或分页优化）。

## 四、 内存泄漏隐患
影响小程序长时间使用的稳定性，易导致越用越卡或闪退：
10. **页面销毁时未及时移除 `wx.on` 类型监听器**（未在 `onUnload` 中成对调用 `wx.off...`，导致内存中残留无效的监听器对象）。
