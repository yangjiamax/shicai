# 识为鲜 V1 - 痛点三修复与落地计划 (SOP)

## 核心定位与背景
针对上线后迭代计划中的**痛点三：购物清单“结束采购”按钮无响应（幽灵按钮 Bug）**，本质问题是用户在“我的 - 历史记录”中打开状态为 `active`（进行中）的清单详情时，页面 (`pages/history-list`) 缺少或遗漏了与 `pages/list` 一致的结束采购逻辑。这导致了核心闭环链路阻塞。

本计划遵循**“规则 → 方案 → 实现 → 验证 → 沉淀”**的 SOP 进行梳理，并在执行前达成一致。

---

## 1. 规则层结论（产品逻辑与体验判断）
- **核心问题**：用户决策路径断层。用户在主清单页勾选完食材后，可能切出去了，之后从“我的-历史记录”重新点开这个“进行中”的清单，希望能直接结束它。
- **职责边界**：`history-list`（历史记录详情）不仅需要展示已完结的数据，对于 `active` 状态的数据，应该承担与 `pages/list` 相同的“结束采购”能力，避免用户被困在详情页无法闭环。
- **约束原则（防踩坑）**：由于结束采购涉及到状态更新和清理，必须采用复用 `dbUtil` 统一更新的方案，确保前后端数据状态一致。结束成功后，需立刻返回上一页（`pages/my`），通过 `onShow` 刷新历史记录列表。

---

## 2. 实现层落地（技术推导与代码方案）
本次改动坚持“最小侵入原则”：
1. **页面层 (`pages/history-list/index.wxml`)**
   - 检查底部 `finish-btn-container` 的渲染逻辑。
   - 补充 `active` 状态下的“结束采购”按钮渲染（复用原有的 `btn-primary` 样式，绑定 `bindtap="finishPurchasing"`）。
2. **数据与逻辑层 (`pages/history-list/index.js`)**
   - 新增 `finishPurchasing` 方法（参考 `pages/list/index.js` 的成熟逻辑）。
   - 流程：
     - 弹出二次确认框（复用全局 i18n 话术）。
     - 调用云端指令，将当前清单状态（以及当前用户的所有 active 脏数据）更新为 `completed`。
     - 成功后，执行 `wx.navigateBack({ delta: 1 })`，平滑退回“我的”页面，不再强行 reload 当前页，把刷新动作交给上一页的生命周期。

---

## 3. 验证与风险（验收闭环）
- **前提假设**：老数据与新数据的状态机只有 `active` 和 `completed`，兼容性良好。
- **验证路径（SOP 要求）**：
  1. 在开发者工具/真机上，将一个清单置为 `active`。
  2. 前往“我的 - 历史记录” -> “账单（汇总选购）”，找到该 `active` 的清单。
  3. 进入 `history-list` 详情页，点击底部的“结束采购”按钮。
  4. 确认是否弹出“确认结束”对话框。
  5. 确认后，是否成功退回上一页，并且该清单状态变为 `completed`。
  6. 控制台无任何 `undefined is not a function` 等红字报错。

---

## 4. 可沉淀的长期协作规则 (Skill/SOP)
- **多入口状态统一**：凡是涉及“核心状态变更”（如“结束采购”）的业务动作，如果存在多入口（如主 tab 页和历史详情页都能访问该数据），必须确保所有入口的事件绑定与处理逻辑一致，防止“幽灵按钮”或“只可看不可动”的死胡同。
- **统一降维回退**：对于在子页面终结了当前数据生命周期的操作，最佳实践是 `wx.navigateBack` + 上一页 `onShow` 静默刷新，而不是在子页面内强刷状态，这更符合用户的自然空间回退预期。
