# 识为鲜 V1 痛点二修复计划：分享裂变链路打通 (Share & Fission Fix Plan)

## 1. 背景与核心问题分析

在小程序正式上线后，用户反馈在尝试分享具体的识物结果、菜谱或购物清单时，接收者点击卡片会被强制跳回首页，导致“所见非所得”，严重阻碍了小程序的裂变增长。

**根本原因（两条断层）：**
1. **分享参数丢失**：在 `pages/result`, `pages/recipe`, `pages/list`, `pages/history-list` 页面中，`onShareAppMessage` 被硬编码死写为 `path: '/pages/index/index'`。分享卡片生成时丢失了真实的页面路径和资源参数。
2. **新用户承接断层**：即使路径正确，新用户点击卡片进入详情页时，会被拦截并跳转至授权页（`pages/onboarding/index`）。但在授权完成后，当前代码逻辑固定调用 `wx.switchTab({ url: '/pages/index/index' })`，导致用户授权后依然回到了首页，遗忘了最初被分享的内容。

## 2. 修改计划与具体落地步骤

为了实现**“老用户所见即所得，新用户授权后顺滑衔接”**的裂变闭环，计划分四步进行改造：

### 第一步：重写各核心页面的分享路牌（改写 `onShareAppMessage`）
让每个页面在分享时，带上真实的自身路径和数据行李：
- **识物结果页 (`pages/result/index`)**：
  - 分享路径带上核心识别结果与标识 `?data={ingredientName: xxx, imagePath: xxx, ...}&shared=1`。
- **菜谱详情页 (`pages/recipe/index`)**：
  - 将当前菜谱名称、核心佐料列表打包成 JSON，附在路径中 `?data={recipeName: xxx, ...}&shared=1`。
- **购物清单页 (`pages/list/index` & `pages/history-list/index`)**：
  - 无论是活跃清单还是历史清单，统一将分享路径指向**历史清单页**，带上 `?listId=xxx&shared=1`（为了安全，被分享者只能以只读的历史清单模式查看，不能直接修改他人的活跃清单）。

### 第二步：打通新用户授权记忆（引入 `redirectUrl` 中间件机制）
实现“哪里来，回哪里去”的顺滑体验：
1. **详情页拦截（入口卫士）**：
   在详情页的 `onLoad` 中，如果检测到用户未完成 onboarding（新用户），则将当前页面的完整路径（包含所有参数）编码为 `redirectUrl`，并跳转至 `/pages/onboarding/index?redirectUrl=...`。
2. **授权页放行（出口路由）**：
   改造 `pages/onboarding/index`，在用户完成资料填写并点击“开启寻鲜之旅”后：
   - 检查 `options` 中是否携带了 `redirectUrl`。
   - 如果有，使用 `wx.redirectTo` 跳转回该目标页面。
   - 如果没有，按原有逻辑 `wx.switchTab` 回首页。

### 第三步：解决分享数据的越权读取（需控制台物理操作）
> **⚠️ 关键阻断**：微信云开发数据库默认权限为“仅创建者可读写”。当用户 B 拿着用户 A 分享的 `listId` 去查询数据库时，会被底层拦截。

**落地方案：**
- 由开发者手动在微信开发者工具的“云开发控制台”中，将 `shopping_lists` 和 `histories` 两个集合的数据权限修改为 **“所有用户可读，仅创建者可读写”**。因 `_id` 为强随机字符串，不知道 ID 无法遍历，安全性可控。

### 第四步：分享视图下的交互与数据归属隔离（防止串台与防懵逼）
当用户 B 打开用户 A 分享的页面时，必须保证所有后续的写入动作（收藏、加入清单）都**完全归属于用户 B 自己**，绝不能污染用户 A 的数据。同时，要解决**被分享者返回上一页无路可走**的体验断层问题。

- **1. 返回路径修复（防懵逼机制）**：
  - **痛点**：微信小程序中，如果是通过分享卡片直接打开的内页（如 `/pages/recipe/index`），左上角默认是“小房子”图标或者根本没有返回按钮。此时调用 `wx.navigateBack()` 会报错，用户看完详情后“无路可走”，只能杀掉小程序。
  - **解法**：在所有详情页（result/recipe/history-list）中，如果检测到是从分享进入（`shared=1` 或页面栈只有1层），必须将页面上的“返回/重试/再拍一次”按钮的行为，从 `wx.navigateBack()` 降级重写为 `wx.reLaunch({ url: '/pages/index/index' })`，强行建立一条回到主线（首页）的退路。
- **2. 识物结果页（收藏动作隔离）**：
  - 检测到 `shared=1` 参数时，前端在渲染前强制清除携带的 `historyId`，并强制将“收藏”按钮状态重置为 `false`（未收藏）。
  - 当用户 B 点击“收藏”时，系统会在数据库中为用户 B 创建一条全新的 `histories` 记录，`_openid` 自动绑定为用户 B。
- **3. 菜谱详情页（加入清单隔离）**：
  - 当用户 B 点击“加入购物清单”时，由于我们调用的是云端 `dbUtil.addIngredientsToList`，该方法会自动寻找或创建**属于用户 B 的活跃清单**，并将食材写入，完美隔离。
- **4. 历史清单页（整体搬运隔离与只读锁定）**：
  - **只读锁定**：对于历史清单页，只要是从分享进来的（`shared=1` 或并非自己的 `listId`），必须强制将 `listStatus` 视为只读状态（例如锁定为 `'completed'` 或 `'shared'`），**彻底隐藏或禁用**单点食材状态的打勾/取消打勾功能，防止用户 B 以为能替用户 A 去买菜。
  - **整体搬运**：只保留“全部加入新清单”按钮。当用户 B 点击时，系统提取页面上的食材纯文本数据，写入到**属于用户 B 的活跃清单**中。
- **5. 菜谱红心收藏状态隔离**：
  - 在清单页和结果页中，如果显示了菜谱（带有红心收藏按钮），该红心的状态（是否点亮）必须根据**当前登录用户（用户 B）的 `recipes` 集合**去实时查询，绝不能沿用分享者（用户 A）的收藏状态。
- **6. 云端图片生命周期防碎（Image Expiration 防护）**：
  - **痛点**：识物结果页分享时，如果带的是云存储的临时连接（或未经长久固化的 `fileID`），接收者打开卡片时可能会因为图片链接过期、被清理或无权读取而导致“白切鸡变成白板图”。或者分享的是微信本地临时路径 `wxfile://` 或 `http://tmp`，接收者绝对打不开。
  - **解法**：在分享之前（如 `onShareAppMessage` 中或生成 `data` 时），如果检测到当前图片是本地临时路径，必须**阻断分享**或**静默上传**。由于同步 API 不支持 await 上传，必须保证**识物结果页在加载完成/识别成功时，立刻自动将图片上传到云存储**（例如存入一个专用的 `shares/` 目录并设置为公有读），分享卡片中只携带持久化的云端 `fileID` 或 `HTTPS URL`。

## 3. 验收标准与验证路径

修改完成后，必须通过以下场景验证方可上线：

### 验收 1：分享参数完整性验证
1. 在开发者工具中，分别进入 `result`, `recipe`, `list` 页面。
2. 点击右上角“发送给朋友”。
3. 检查控制台或生成的 ShareTicket，确认 `path` 参数不再是 `/pages/index/index`，而是包含了当前页面的真实路径和参数（如 `/pages/recipe/index?data=...&shared=1`）。

### 验收 2：老用户（已授权）与数据隔离验证
1. 在开发者工具的“编译模式”中，添加自定义编译条件，带上 `shared=1` 及具体数据进入识物结果页/清单页。
2. **预期结果 1（收藏隔离）**：页面底部“收藏”按钮必须是**未高亮**状态。
3. **预期结果 2（写入隔离）**：点击“收藏”或“加入清单”，能正常成功，并在数据库中查看到这是当前测试账号（用户 B）新创建的数据，原分享者（用户 A）的数据未被修改。
4. **预期结果 3（防懵逼）**：点击页面左上角或页面内的“返回”类按钮，能顺滑跳转回首页，不发生无响应或报错。
5. **预期结果 4（只读锁定）**：分享进入他人的历史清单页时，点击单个食材不应有任何划线/打勾反应。
6. **预期结果 5（图片不碎）**：使用另一台手机（非拍摄者）扫码打开分享页，能够清晰看到原本被识别的实物图片，而不是加载失败的图标。

### 验收 3：新用户（未授权）场景模拟（裂变核心）
1. 在开发者工具中点击“清空全部缓存”。
2. 保持上述的自定义编译条件（带分享参数的详情页路径）重新编译。
3. **预期结果**：
   - 启动后，系统瞬间拦截并跳转至 onboarding 授权页。
   - 授权页的 URL 中应能看到附带的 `redirectUrl` 参数。
   - 用户正常完成头像昵称等资料填写。
   - 点击“保存”或“开启寻鲜之旅”后，**不回首页**，而是自动重定向到最初被分享的详情页，并成功展示内容。

## 4. 风险评估与回滚策略
- **输入输出守恒**：老用户的正常浏览、正常分享不受任何影响。未带 `redirectUrl` 的正常注册依然回首页。
- **极端弱网/参数解析防错**：在解析分享的 `data` JSON 字符串时，加入 `try-catch` 兜底。若解析失败或数据损坏，优雅退回首页并提示“内容已失效”，绝不引发白屏。
- **回滚策略**：若改动引发不可预知路由死循环，可通过 Git 瞬间回退 `onboarding/index.js` 和各页面的 `onShareAppMessage`，不涉及核心数据库结构变更。
