# 识为鲜 V1 上线后核心问题修复与迭代计划 (Post-Launch Safe Iteration Plan)

## 背景与目标
小程序正式上线后，面临两大核心痛点，直接影响用户留存（性能）与裂变增长（分享）。本计划旨在通过“小步快跑、极致防踩坑”的原则，逐一攻克这两个阻塞性问题。

## 核心痛点记录

### 痛点一：大模型识别环节性能与兼容性极差（流失元凶）
**表现：**
1. **转化速度极慢**：拍照识别、文本转清单、语音转清单处理极慢，经常超过 60 秒导致系统宕机（云函数超时或前端请求超时）。
2. **环境兼容性差**：弱网环境或境外使用时，大模型服务经常调用失败。

**根本原因与落地方案：**
- **超时设置冲突**：云函数执行时间上限配置（`config.json`）仅为 60s，但前端代码或云函数内 Axios 请求超时设为了 90s。网络较差时，云函数会先被系统强制掐断。
- **完全同步阻塞**：目前没有使用大模型的流式（Streaming）输出，也缺乏弱网降级模型（Fallback）策略，用户只能死等，极易白屏或请求断掉。

**验收标准（验证路径）：**
1. **开发者工具验收**：使用 Network Throttling 模拟 `Slow 3G`，发起识图/录音请求，验证请求能否在 60s 内得到明确处理（不论成功还是兜底报错，不再出现 60s 后的底层宕机断联）。
2. **真机断网测试**：在开启飞行模式或极弱网下发起请求，确认有友好的 Fallback 提示，而不是一直转圈死等。

### 痛点二：分享页面跳转逻辑断层（阻碍裂变）
**表现：**
1. **所见非所得**：用户分享具体内容（食材结果、菜谱详情、购物清单），接收者点击卡片后却被强制跳回首页，无法直接看到被分享的内容。

**根本原因与落地方案：**
- **路由硬编码**：核心页面（`pages/result`, `pages/recipe`, `pages/list`）的 `onShareAppMessage` 被硬编码写死了 `path: '/pages/index/index'`，导致分享卡片丢失了具体的资源 ID 和路由，接收者必然只能回到首页。需改写分享参数并处理首页的重定向承接。

**验收标准（验证路径）：**
1. **同账号分享验收**：开发者工具中点击这 3 个核心页面的右上角分享，查看生成的 `shareTicket` 中的 `path` 参数是否包含了当前的真实路径与 ID（如 `/pages/result/index?id=xxx`）。
2. **场景模拟验收**：在 IDE 的“编译模式”中，手动将启动页面和参数设为刚才生成的分享参数。
   - **老用户模拟**：直接进入对应详情页（所见即所得）。
   - **新用户模拟**：清空缓存后启动，需先跳转授权页，授权完毕后自动重定向回被分享的详情页，不回首页。

### 痛点三：购物清单“结束采购”按钮无响应（用户反馈 Bug）
**表现：**
1. **主流程阻塞**：用户在反馈控制台指出：“把推荐的食材都勾掉了，但点击结束采购按钮没有反应”。这属于核心闭环链路上的严重 Bug，导致用户无法正常完结当前购物清单。

**根本原因与落地方案：**
- **事件绑定缺失（幽灵按钮）**：在“我的-集邮册（历史记录）”中打开进行中（active）的清单时，`pages/history-list/index.wxml` 渲染了“结束采购”按钮，但 `history-list/index.js` 中根本没有定义 `finishPurchasing` 这个方法，导致点击事件被直接抛弃，无任何响应。

**验收标准（验证路径）：**
1. **测试用例**：去“我的 - 历史记录”中，找一条状态为 `active` 的清单并进入详情。
2. **勾选反馈**：勾掉任意（包含普通和推荐）食材。
3. **按钮响应**：点击底部“结束采购”，能够弹出“确认结束”的二次确认框，确认后成功跳转至“已完结（completed）”状态界面并返回上一页刷新列表，控制台不报任何 undefined is not a function 错误。

### 痛点四：右滑录音按钮后，偶发录音功能无法调用并持续报错（隐性/偶发 Bug）
**表现：**
1. **偶发性交互冲突导致功能失效**：该问题并非必现，多数情况下右滑操作正常可用。但在某次特定场景（尝试多次右滑）下，录音功能彻底失效并持续报错，且目前难以稳定复现。

**根本原因与落地方案：**
- **状态机死锁（竞态条件）**：在快速滑动、多次滑动或滑动出界触发 `touchcancel` 的极端时序下，前端 `onSwipeEnd` 触发了多次录音启动，但底层的 `recorderManager` 或者前端维护的 `isRecording` 锁并未正确释放。微信底层拒绝连续并发启动录音，导致状态机彻底锁死。必须在所有滑动打断/取消的回调中加上强制的锁重置与 stop 调用。

**验收标准（验证路径）：**
1. **暴力滑动测试**：在 IDE 和真机上，对着悬浮麦克风疯狂上下左右滑动，并迅速松手。重复 20 次。
2. **中断测试**：按下录音的瞬间让手指滑出屏幕边缘（模拟 `touchcancel`），随后再次点击麦克风。
3. **通过标准**：任何异常滑动后，再次点击麦克风时，不会出现“录音初始化失败 / 当前正在录音”的控制台红字报错，能正常发出“滴”声并开启新一轮录音。

### 痛点五：重新打开小程序时状态滞后，误弹“开启云存储”提示（用户体验阻断）
**表现：**
1. **状态恢复迟缓**：杀掉小程序重新打开时，有一小段时间系统会认为用户是“未注册/未授权”状态，甚至弹出要求“开启云存储/重新授权”的提示，但过一会儿（几秒后）又自动恢复正常。
2. **资料更新滞后**：修改头像或填写个人信息后，页面显示存在明显的延迟。

**根本原因与落地方案：**
- **生命周期并发漏洞**：首页和“我的”页面的 `onShow` 生命周期中遗漏了对全局鉴权锁（`await app.authReadyPromise`）的等待。杀掉重开时，云端静默登录请求还在飞，页面就用空身份开始查数据了，导致误判为“游客”并弹出授权提示。
- **缺乏乐观更新**：修改资料后还在等云端拉取结果。需要加入 Optimistic UI 策略，提交即刷新本地缓存并渲染。

**验收标准（验证路径）：**
1. **冷启动模拟**：使用 IDE 右上角的“清理全部缓存与授权数据”，然后重新编译。登录成功后，杀掉微信后台（真机）或 IDE 重新编译（热启动/冷启动交替）。
2. **通过标准 1（不闪弹窗）**：页面保持骨架屏或白屏几百毫秒，直到左上角头像渲染出真实数据后，才显示内容，期间绝对不出现“请开启云存储”的误判弹窗。
3. **通过标准 2（资料秒切）**：进入“编辑资料”，更换头像后点击保存，页面退回到“我的”界面，头像在 0.1s 内肉眼可见地变为新头像，无需下拉刷新。

### 痛点六：极端复杂图片压缩后依然触发 data exceed max size 导致调用失败（隐性 Bug）
**表现：**
1. **极限场景崩溃**：遇到细节极其丰富、噪点或纹理极多（如这盘带有高频纹理的白切鸡）的图片时，固定压缩率（如 0.8）生成的 Base64 字符串依然会超过微信云函数 1MB 的硬性 Payload 限制，导致 `cloud.callFunction:fail`。

**根本原因与落地方案：**
- **写死的压缩率与 Payload 限制冲突**：当前 Canvas 离屏压缩逻辑固定了 `0.8` 的 JPEG 质量。对于极度复杂的图片，即便压到 0.8，其体积加上 Base64 转换的 33% 膨胀，依然会突破微信云函数 1MB 的天花板。必须在前端引入“体积校验与动态递归压缩”机制（即判断如果 >1MB 就递归降质到 0.6 再测），或改为“先传云存储临时文件，再传 fileID 给云函数”的绕行架构。

**验收标准（验证路径）：**
1. **极限样张测试**：必须使用你提供的报错样张 `d:\第二曲线\识为鲜PinFresh\debug_and_optimization\73ec34bf09ebfc7cb651a6015f702e33.jpg`（白切鸡高频纹理图）进行上传测试。
2. **日志观测**：在控制台中必须能看到“递归压缩触发：从 0.8 降级到 0.6”的降级日志，且最终生成的 payload `.length` 被成功拦截在 `1,000,000` 以内。
3. **闭环成功**：点击拍照/相册上传该图，能正常拉起云函数并返回识别清单，不再出现 `data exceed max size` 报错红框。

### 痛点七：微信数据分析平台与数据库用户数存在巨大差异（数据统计漏斗断层）
**表现：**
1. **数据严重不一致**：在云开发数据库的 `users` 集合中仅看到不到 10 个用户文档，但在微信小程序官方数据分析平台上却显示有 60 多个累计访问用户，存在巨大差值。

**根本原因分析（为什么会出现差值？在哪里？）：**
1. **统计口径的物理差异**：微信官方分析平台统计的是“独立访客（UV）”，只要用户点开了小程序（哪怕只看了一眼白屏就退出），微信底层就会记录一个活跃 OpenID。
2. **业务落库时机严重滞后（核心原因）**：当前的业务逻辑采用了“延迟注册/表单驱动注册”的模式。用户很可能只有在完成 Onboarding（填写国籍、头像昵称等）或者进行实质性操作（如保存清单）时，才会向 `users` 集合写入数据。
3. **首屏流失率高（漏斗流失）**：这意味着有 50 多位用户在打开小程序后，在真正触发“写入数据库”的动作之前就离开了（例如被强制的 Onboarding 流程劝退，或因为网络慢加载白屏而跳出）。

**如何改进（落地方案）：**
1. **前置“静默影子账户”落库（Immediate Silent Registration）**：修改 `app.js` 的鉴权逻辑。在 `login` 云函数成功换取到 `OpenID` 的那一刻，立刻通过后端静默执行一次 `upsert`（存在则更新登录时间，不存在则创建），生成一个基础的用户文档（仅包含 `_openid`, `createdAt`, `lastLoginAt`, `role: 'visitor'`），将落库动作与业务表单解耦。
2. **用户身份状态分级**：在 `users` 集合中引入状态机机制，区分 `visitor`（仅访问未填资料）、`onboarded`（已填资料正式用户），从而精准分析转化率。
3. **建立关键节点漏斗埋点**：在微信后台配置自定义事件（如 `App_Launch` -> `Login_Success` -> `Onboarding_Show` -> `Onboarding_Complete`），精准定位这 50 多个用户到底是在哪一个环节（授权弹窗、选国籍、还是纯粹的加载慢）流失的。
4. **体验优化**：若数据证明 Onboarding 是流失元凶，则需将非必要的资料收集（如选国籍/饮食偏好）从冷启动拦截流程中剥离，改为“先体验核心识物功能，后置引导完善资料”。

### 痛点八：大模型识物容错率低，缺乏手动纠偏机制（容错与体验问题）
**表现：**
1. **识别错误无法修改**：由于大模型（如视觉识别）并非 100% 可靠，有时会将食材识别错误。当前用户在得到错误的识别结果后，只能无奈接受或重新拍照，缺乏手动修正的途径。

**根本原因与落地方案：**
- **缺乏人工介入环节**：目前的单向流（图片 -> AI -> 结果呈现）没有给用户提供介入修改的入口。
- **落地方案**：在识物结果页（或清单生成页）的食材名称旁边，增加一个“编辑（如铅笔图标）”按钮。当用户发现 AI 识别有误时，可以点击该按钮，通过弹窗或内联输入框手动修改/输入正确的食材名称，确保最终加入清单或生成食谱的食材数据是准确的。

**验收标准（验证路径）：**
1. **交互验证**：在识物结果页，识别出的食材名称旁正确显示编辑（铅笔）图标。
2. **修改验证**：点击编辑图标后，可手动输入正确名称并保存。保存后，当前页面的食材名称立即更新，且后续的“查看食谱”和“加入清单”等动作均使用用户修改后的正确名称。
