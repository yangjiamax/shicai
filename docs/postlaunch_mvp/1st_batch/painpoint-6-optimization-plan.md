# 痛点六：极端复杂图片防超限与极致压缩优化方案 (Pain Point 6 Optimization Plan)

## 1. 核心问题与背景
- **痛点**：在处理高频纹理图（如带有复杂噪点的白切鸡样张）时，即使用离屏 Canvas 按照 0.8 质量压缩，生成的 Base64 字符串依然会超过微信云函数 1MB 的 Payload 物理限制，导致 `data exceed max size` 崩溃。
- **现状**：当前的前端压缩逻辑是“盲盒式”的，缺乏对压缩后真实体积的感知与拦截。一旦超限，用户只能面临死锁或白屏，没有任何降级路由。

## 2. 核心原则与判断 (规则层)
根据“后上线时代代码演进法则”与“自带兜底方案”的协作原则，本方案设计遵循：
- **极度安全感（不把天花板交给运气）**：前端必须拥有对 Payload 体积的 100% 掌控权，必须在调用云函数前进行长度预判。
- **纯前端闭环（轻量化原则）**：无需引入云存储中转的复杂架构，完全通过前端 Canvas 动态递归降质来保证最终输出体积达标，避免增加云端清理逻辑与存储成本。
- **体验优先**：绝大部分常规图片一次性 0.8 质量通过，仅针对极少数极端图片执行降质，以最小的性能开销换取 100% 的可用性。

## 3. 落地方案：前端动态双维降级兜底 (实现层)
本方案采用“一次清洗，双维降级”的纯前端闭环策略。

### 核心执行逻辑
- **预判水位**：计算 Base64 字符串长度。设定安全水位为 `850,000` 字节（约 850KB，给 HTTP Header 及其他 JSON 字段留出缓冲）。
- **第一级：标准离屏清洗**：使用 `wx.createOffscreenCanvas` 剥离 EXIF 隐藏数据，以 `0.8` 质量导出 Base64。若 `< 850,000`，直接发起云函数请求（最快链路）。
- **第二级：动态降画质（主要对 PC/安卓有效）**：
  - 若 0.8 压缩后长度 `>= 850,000`，无需重新绘制 Canvas，通过循环以步长 `0.2` 递减导出质量。
  - 由于 iOS 微信底层的 `toDataURL` 存在 quality 参数失效的已知 Bug，此阶段在 iOS 上可能无法有效降低体积。
- **第三级：动态降分辨率重绘（iOS 终极兜底，替代黑白方案）**：
  - 若画质降到 `0.2` 体积依然超限（iOS 必现），触发终极兜底：缩小画布宽高（每次缩小 40%），清空并重新 `drawImage` 绘制。
  - **为什么不用黑白（灰度）？**：食物的颜色（红润的肉、翠绿的菜）是大模型识别种类和评估新鲜度的核心特征，黑白化会引发严重的 AI 幻觉和准确率暴跌。而视觉大模型底层实际输入分辨率很小（通常 224x224 到 448x448），所以**缩小图片分辨率不仅能断崖式降低 Base64 体积，且对 AI 识别完全无损**。
- **日志输出**：若触发降质，需在控制台打印 `[ImageCompress] 递归降质触发: 初始 0.8 -> 最终 xx，最终体积 xx KB`。

## 4. 验收标准与验证路径
1. **极限样张测试**：必须使用指定报错样张 `d:\第二曲线\识为鲜PinFresh\debug_and_optimization\73ec34bf09ebfc7cb651a6015f702e33.jpg` (白切鸡高频纹理图) 进行真机/开发者工具测试。
2. **日志观测**：
   - 控制台必须清晰打印路由轨迹日志，能看到“递归降质触发：从 0.8 降级到 0.6 (或更低)”的降级日志。
   - 最终生成的 Payload 长度必须被成功拦截在 850KB 水位线以内。
3. **闭环成功**：
   - 彻底消灭 `data exceed max size` 报错红框。
   - 点击拍照/相册上传该图，能正常拉起云函数并返回识别清单。

## 5. 长期可沉淀规则 (SOP)
- **前端 Payload 防超限范式**：在小程序端向云函数传递图片等大体积数据时，**必须标配“本地内存体积校验 + 动态递归降质”机制**，不再绝对信任固定的压缩率。该模式可通过纯前端闭环极大地提升上传稳定性，且无云端改造成本。
