# 知食角小程序性能优化专项规划

> **文档状态**：初步规划
> **背景**：基于 2026-04-16 的 MiniTest 智能化 Monkey 测试报告（综合得分 82，存在 10 项待改进项）。
> **目标**：从底层解决包袱过重、通信浪费、渲染卡顿和内存泄漏等隐患，特别提升中低端机型（如 iPhone 11）的运行流畅度。

本规划独立于日常业务功能迭代（Iteration Plan），作为**技术债治理专项**，建议在业务需求平缓期穿插执行。

---

## 一、 启动与首屏性能治理（目标：秒开体验）

**【现象描述】**
* 页面 `App.onLaunch` 前存在耗时操作（如 `app.js` 中的全局代码）。
* `App` 生命周期执行耗时过长。
* 启动阶段重复获取相同 key 的 storage 信息。
* 首屏渲染前多次调用同步 API。

**【根本原因分析】**
1. **全局作用域阻塞**：在 `app.js` 的顶层（即不在任何生命周期函数内的全局代码）或 `onLaunch` 中，滥用了同步 API（如 `wx.getStorageSync`、`wx.getSystemInfoSync`）。这些同步方法会直接阻塞主线程，导致首屏 UI 迟迟无法开始渲染。
2. **Storage 滥用**：为了方便，代码中可能多处直接调用 `wx.getStorageSync('userInfo')` 或 `wx.getStorageSync('lang')`，导致同一个 key 在启动瞬间被读取多次。本地 I/O 读写虽然快，但多次叠加依然会产生可观的耗时。

**【改进规划 / Action Items】**
* [ ] **建立全局内存缓存机制**：在 `app.js` 的 `globalData` 中建立 Storage 的内存映射。应用启动时异步读取一次 Storage 并存入 `globalData`，后续所有页面均从 `app.globalData` 读取，修改时同步更新内存与 Storage。
* [ ] **剥离 `app.js` 耗时操作**：排查 `app.js` 顶层及 `onLaunch`，将非首屏必须的初始化逻辑（如预拉取非关键配置、清理过期缓存）推迟到首屏渲染完成后的 `onShow` 中，或使用 `setTimeout` 放入宏任务队列。
* [ ] **同步 API 替换**：将非必须的 `Sync` 后缀 API（如 `wx.getSystemInfoSync`）替换为异步版本，或确保其只执行一次并缓存结果。

## 二、 数据通信与逻辑层瘦身（目标：丝滑跳转与交互）

**【现象描述】**
* `setData` 传入未绑定在 WXML 的变量。
* 页面渲染前多次调用同步 API。
* 普通运行阶段重复获取相同 key 的 storage 信息。
* 页面跳转响应慢（低端机尤其明显）。

**【根本原因分析】**
1. **`setData` 黑洞**：小程序逻辑层（JS）和视图层（WXML）是分离的，`setData` 是它们之间唯一的通信桥梁。如果把后端返回的巨大 JSON（如整个大模型解析结果、包含数十个字段的食谱详情）直接 `setData`，但 WXML 只用到其中 3 个字段，会导致极大的序列化/反序列化性能浪费。
2. **跳转阻塞**：目标页面的 `onLoad`/`onShow` 中存在耗时的同步任务，或者刚进页面就立刻执行了巨大的 `setData`，导致页面转场动画卡顿，给用户“点击无响应”的错觉。

**【改进规划 / Action Items】**
* [ ] **`setData` 纯净度审计**：全局排查 `setData` 调用。对于不需要在页面上展示的数据（如内部状态标记、用于发请求的 ID、全量原始数据等），直接赋值给 `this.xxx` 或 `this.data.xxx`，绝对禁止放入 `setData`。
* [ ] **页面跳转预加载**：对于需要传递大量数据的页面跳转（如从结果页跳食谱页），优先使用 `wx.navigateTo` 的 `EventChannel` 进行通信，而不是通过 Storage 中转或 URL 拼接 JSON。
* [ ] **骨架屏与异步渲染**：跳转新页面时，先 `setData` 极少量的基础数据（撑起骨架屏），等页面转场动画结束（或放到 `onReady` 中）再执行复杂的业务计算和全量渲染。

## 三、 视图层渲染压力释放（目标：长列表不卡顿）

**【现象描述】**
* 过大的 WXML 节点数目。

**【根本原因分析】**
随着用户使用时长增加，历史记录、购物清单、或长篇的食谱详情页节点激增。如果将几百条数据一次性渲染到 WXML 中，会极大增加 WebView 的内存占用和排版计算耗时，导致滑动掉帧。

**【改进规划 / Action Items】**
* [ ] **长列表分页改造**：排查历史记录页（`history-list`）等可能无限增长的列表，确保后端数据库查询已开启基于游标（`skip`/`limit`）的分页拉取，前端配合 `onReachBottom` 进行上拉触底追加。
* [ ] **引入虚拟列表（可选，视严重程度）**：如果存在单页必须展示超长列表的需求，可引入 `recycle-view`（虚拟列表），只渲染屏幕可视区域及其前后的少量节点。
* [ ] **组件级按需渲染**：对于默认隐藏的复杂弹窗、抽屉等组件，使用 `wx:if` 而不是 `hidden`，确保在不需要展示时不创建其 WXML 节点。

## 四、 内存与稳定性排雷（目标：久用不闪退）

**【现象描述】**
* 页面销毁时未及时移除 `wx.on` 类型监听器。

**【根本原因分析】**
在页面 `onLoad` 或 `onShow` 中注册了全局事件（如 `wx.onThemeChange` 监听暗黑模式、`wx.onNetworkStatusChange`、或自定义的 EventBus 监听），但在 `onUnload` 时没有成对调用对应的 `wx.off...` 方法。导致每次进入该页面都会新增一个监听器，不仅重复执行逻辑，还会导致页面实例被闭包引用，永远无法被垃圾回收（内存泄漏）。

**【改进规划 / Action Items】**
* [ ] **全局事件防漏审计**：全局搜索 `wx.on`、`EventBus.on` 等监听绑定代码。
* [ ] **配对销毁机制**：强制规范：凡是在组件 `attached`/页面 `onLoad` 中绑定的全局监听，必须在 `detached`/`onUnload` 中进行 `wx.off...` 解绑。
* [ ] **单例模式替代**：对于某些只需全局监听一次的状态（如网络状态），考虑移入 `app.js` 统一监听，然后通过状态分发，避免在各个页面中反复绑定。

## 五、 建议实施策略

由于性能优化属于非功能性需求，建议按照**“投入产出比（ROI）”**分步实施：

1. **Quick Wins (见效快、改动小) - 建议本周/下周穿插完成**：
   * 修复 `wx.on` 内存泄漏问题（全局搜索排查，加上 `off` 即可，风险极低）。
   * 审计核心页面的 `setData`，剔除明显不需要渲染的冗余大对象。
2. **中期改造 (需要局部重构) - 建议纳入下一个 Minor 版本**：
   * `app.js` 启动瘦身，建立 Storage 内存缓存机制。
   * 历史记录等长列表的分页加载补全。
3. **长期观察 (持续监控)**：
   * 建立性能体验基线，未来每次 Major 版本发布前，执行一次 MiniTest，确保性能评分不低于 85 分。
