# 《知食角》V2.0 开发计划 (Dev Plan)

> **目标**：将应用从单线工具升级为“识物百科 + 智能购物管家”双引擎。
> **策略**：自底向上，先解决数据结构和云端 AI 能力（造引擎），再处理前端数据流转（修管道），最后完成 UI 重构与交互优化（做车身）。

## 阶段一：底层基建与 AI 能力（M1）
> **目标**：确立 Shopping List 与 Ingredient 的数据结构，跑通语音/文本提取清单的 AI 能力。

*   **[M1.1] 数据库结构升级**
    *   在云数据库中新建 `shopping_lists` 集合，字段参考 PRD 4.0。
    *   在云数据库中新建 `ingredients` 集合，设计与 `list_id` 的关联。
    *   **✅ 验收标准**：能在云开发控制台看到 `shopping_lists` 和 `ingredients` 集合，且能成功执行基础的增删改查联表操作。
*   **[M1.2] 云函数：清单提取能力 (AI 结构化)**
    *   新建云函数 `extractList`。
    *   编写并调优 Prompt：输入一段自然语言（“我要买两斤排骨和一头蒜做糖醋排骨”），输出结构化的 JSON 数组（含 `standard_name`, `quantity`, `unit`, `category`, `source_recipe`）。
    *   处理兜底逻辑（如未指定菜谱时 `source_recipe` 为空）。
    *   **✅ 验收标准**：云函数可独立测试运行，输入测试文本能稳定返回格式正确的 JSON 数组，兜底逻辑（无菜谱、无法识别分类等）不报错。
*   **[M1.3] 语音转文本能力接入**
    *   在小程序端直接调用微信原生录音接口接收用户录音，在云端对接大模型API（如Whisper或其他大厂语音转文字API）实现“录音文件 -> 文本提取”链路，不再直接使用微信同声传译插件。
    *   **✅ 验收标准**：在小程序测试页中录入语音，能准确转换为文本，并成功调用 `extractList` 云函数返回结构化数据。

## 阶段二：核心业务逻辑与状态管理（M2）
> **目标**：打通多端点加入清单的逻辑，并在前端实现“双视角”的数据推导。

*   **[M2.1] 活跃清单管理机制**
    *   实现逻辑：检查今日是否已有未完成的购物清单。如果有，则获取其 `list_id`；如果没有，则自动创建一个标题为“YYYY-MM-DD 购物清单”的新 List。
    *   **✅ 验收标准**：每日首次进入触发创建新清单，同日再次进入能正确获取已创建的 `list_id`，不重复创建。
*   **[M2.2] 多端点数据写入接口**
    *   **端点 A（记一笔）**：调用 M1.2 的云函数提取数据后，将结果写入当前活跃清单。
    *   **端点 B（菜谱一键加入）**：修改现有的识物结果页逻辑，将“生成佐料采购单”的动作改为：将菜谱中的佐料格式化为 Ingredient 对象，直接写入当前活跃清单（`source_recipe` 标记为该菜名）。
    *   **✅ 验收标准**：通过“记一笔”和“菜谱一键加入”两个入口操作后，数据库的 `ingredients` 集合中能正确新增关联当前 `list_id` 的数据记录。
*   **[M2.3] 前端双视角状态推导 (核心难点)**
    *   获取当前活跃清单的所有 Ingredient 数据。
    *   编写纯函数：`derivePlanningView(ingredients)`，按 `source_recipe` 进行 groupBy。
    *   编写纯函数：`deriveExecutionView(ingredients)`，按 `category` 进行 groupBy，并对相同 `standard_name` 的项进行数量累加合并。
    *   **✅ 验收标准**：提供 Mock 数据测试这两个纯函数，输出的树形结构/合并结果完全符合预期（同名食材数量正确相加，分类/菜谱归属正确）。

## 阶段三：前端页面与交互平滑升级（M3）
> **目标**：在不破坏 MVP 现有视觉习惯的前提下，平滑融入 V2 的双引擎 UI，完成从“单次工具”到“常驻 Tab”的过渡。

*   **[M3.1] 导航架构与首页微调 (UI 严格缝合)**
    *   **原则**：绝对禁止脱离 `design` 文件夹内的既有 UI 规范闭门造车，采用“渐进式增强”。
    *   **导航升级**：将底栏平滑扩展为 3 个 Tab：“首页(识物)”、“清单(管家)”、“我的(个人中心)”，新增 Tab 必须复用原有 SVG 图标/颜色体系。
    *   **首页保留与融合**：**不可随意修改** MVP 首页“拍照识食材”和“相册上传”的中心大按钮视觉。将“🎤 记一笔”作为平级功能（如左右分栏）或悬浮 FAB（保持原生/高阶感）融入，严禁产生“两套 UI 拼凑”的廉价感。
    *   **首页动态卡片**：将原首页底部的“最近一次”区域替换为“今日清单进度卡片”（显示 x/y 项已购），点击直达清单 Tab。
    *   **✅ 验收标准**：底部出现 3 个 Tab 且可顺畅切换；首页视觉无破坏感，“记一笔”和“进度卡片” UI 协调；进度卡片数据与实际清单同步。
*   **[M3.2] 识物结果页链路改造**
    *   **按钮替换**：将结果页的“生成佐料采购单”按钮修改为“加入今日购物清单”。
    *   **交互优化**：点击后不再 `push` 到单次采购单页，而是弹出 Toast“已加入清单”，并展示“去查看”操作项（跳转至清单 Tab），保障识物心流不断。
    *   **跨文化及多语言功能保留**：改造期间，绝对不能破坏原本的“熟悉的滋味/试试当地做法”双 Tab 逻辑及对应的 i18n 渲染引擎。
    *   **✅ 验收标准**：点击“加入清单”成功弹出 Toast 并写入数据，点击“去查看”能跳转清单 Tab；双 Tab 菜谱及多语言切换功能 100% 正常工作。
*   **[M3.3] 全新“清单”Tab 页 (双视角 UI)**
    *   将旧版 `/pages/list/index` 升级为底部 Tab 页面。
    *   顶部增加 Toggle 开关（“逐道菜选购/按菜谱” vs “汇总选购/按动线”），保持与 MVP 一致的卡片式、圆角 UI 风格。
    *   实现逐道菜选购的 Grouped List 渲染及按菜谱删除交互。
    *   实现汇总选购的 Grouped List 渲染及丝滑打勾动画（修改 status 为 bought）。
    *   **✅ 验收标准**：能流畅切换双视角且数据展示无误；在逐道菜选购删除某菜谱时，相关食材同步移除；在汇总选购打勾时有过渡动画，且状态实时同步到云端。
*   **[M3.4] 气泡式弱提示合并交互**
    *   在前端维护一个相似度检测逻辑（或依赖 AI 返回的标记）。
    *   在列表渲染时，为相似项添加视觉“连线”或“气泡”，采用与 MVP 风格一致的轻量提示框。
    *   实现点击气泡弹出的确认合并交互，并调用云端更新数据。
    *   **✅ 验收标准**：当清单中出现相似食材（如“猪肉”和“五花肉”）时能出现 UI 提示，用户点击确认后能合并成一条数据，数量正确叠加。

## 阶段四：联调测试与 P1 体验增强（M4）
> **目标**：全链路回归，并逐步引入历史视角重构等增强功能。

*   **[M4.1] 全链路回归测试 (含 MVP 防退化)**
    *   测试从语音输入到双视角查看，再到打勾购买的完整闭环。
    *   测试从“扫一扫” -> 看做法 -> 加入清单 -> 回首页查看动态卡片的闭环。
    *   **【关键】MVP 核心资产回归测试**：确保“双轨食谱(熟悉/当地)”、“B站视频获取”、“多语言(i18n)切换”、“带图分享卡片”等 MVP 核心功能在新版本中运作正常，没有任何功能遗漏或退化。
    *   **✅ 验收标准**：上述所有核心链路均能一气呵成走通，无卡顿或报错；MVP 旧有核心功能 100% 测试通过。
*   **[M4.2] 历史双视角重构 (P1 优先级)**
    *   升级“我的”页面中的历史记录。
    *   增加“图鉴模式（集邮）”瀑布流渲染。
    *   增加“账单模式（复购）”时间轴渲染及一键加入当前清单功能。
    *   **✅ 验收标准**：“我的”页面能正常按图鉴和账单两种模式展示历史数据，上拉加载流畅；“一键加入”能成功将历史食材加入今日清单。
*   **[M4.3] 细节打磨**
    *   增加打勾时的震动反馈和解压音效。
    *   优化各个加载态和空状态的 UI/文案。
    *   **✅ 验收标准**：真机测试下，打勾有明确的物理反馈（震动/声音）；网络延迟或无数据时，有符合品牌调性的占位图或 Loading 提示。

## 阶段五：后续技术优化路线 (未来规划)
> **目标**：在核心业务跑通并稳定后，对特定功能的技术实现进行优化升级，降低成本或提升体验。

*   **[M5.1] 语音识别技术路线优化 (同声传译)**
    *   在确保基础录音转文字 API 可靠使用的前提下，作为性能与成本优化的补充方案，评估并重新接入“微信同声传译”等端侧插件。
    *   **预期收益**：实现流式的“边说边转”效果，降低用户的等待延迟；减少云端大模型 API 的调用成本。
    *   **✅ 验收标准**：在小程序管理后台完成插件授权，成功配置并调用插件。在弱网或正常环境下，相比云端 API 具有更低的延迟与更高的稳定性。
