# 识为鲜 PinFresh 正式上线后的安全迭代计划

> 适用背景：小程序已经有正式用户，接下来所有开发都要同时满足两件事——继续修正和迭代、尽量不扰动正式环境与真实用户数据。
>
> 这份文档不是“马上改代码”的清单，而是我们先达成一致的工作方案。等你确认后，再按阶段开始筹备和落地。

## 1. 先说结论

从现在开始，这个项目要从“单人试验式开发”切换成“上线产品式开发”。

最核心的变化只有 4 条：

1. **正式环境不再拿来日常开发和联调**
2. **真实用户数据不再拿来随便测试**
3. **任何会改数据结构、改云函数、改鉴权的动作，都先在测试环境验证**
4. **每次发布都要有备份、检查、回滚方案**

如果只记住一句话，那就是：

**以后默认在测试环境开发，在体验版验收，在正式环境做最小、可控、可回滚的发布。**

---

## 2. 为什么现在必须换开发方式

根据当前仓库状态，已经出现了几个典型风险：

- 小程序前端当前直接写死了单一云环境，说明开发、联调、线上很可能还在混用同一套云资源
- 项目里虽然已经有 `temp_` 集合与迁移逻辑，但它解决的是“游客转正式”的数据归属问题，不等于“开发环境和正式环境隔离”
- 安全审核清单还是未完成状态，说明上线后的运行治理还没有真正建立起来
- 一旦继续直接在正式库上改逻辑、跑测试、调云函数，就可能出现脏数据、误删、重复迁移、用户数据异常

也就是说，**你现在最需要补的，不是某一个功能，而是一整套“上线后继续开发”的护栏。**

---

## 3. 我们接下来采用的总策略

我建议采用下面这套策略：

### 策略 A：三套环境隔离

把云开发环境拆成三层：

- **开发环境 dev**：你自己日常写代码、跑联调、做实验
- **预发布环境 staging**：体验版验收、回归测试、模拟上线前状态
- **正式环境 prod**：只服务真实用户，不做随手测试

### 策略 B：三类数据分开

数据也分成三类看待：

- **开发测试数据**：随便造，允许清理
- **预发布验收数据**：接近真实，但不是正式用户资产
- **正式用户数据**：只允许通过正式发布逻辑产生，不能拿来做验证材料

### 策略 C：先建护栏，再做迭代

先做这些底层保护，再继续功能开发：

1. 环境隔离
2. 发布流程
3. 备份与回滚
4. 数据迁移规则
5. 安全检查清单

### 策略 D：以后所有需求都走同一条路径

以后修一个 bug、加一个功能、改一个页面，都按这个顺序：

**需求确认 → 测试环境开发 → 测试环境验收 → 预发布回归 → 正式发布 → 发布后观察**

## 4. 分阶段计划

我建议不要一下子全面重构，而是分 4 个阶段推进。

## 阶段 1：先止血，建立最基本的安全边界

### 阶段 1：先止血，清理战场与建立最基本的安全边界

#### 目标

先确保后续开发不会继续直接扰动正式环境，同时把乱糟糟的文件夹整理干净。

#### 这一阶段要完成什么

1. **大扫除：清理并归档无用文件**
   - 现在的根目录混杂了备份、设计稿、旧代码。我们要建一个 `_archive`（归档）文件夹，把它们移进去。
   - **要移走的东西**：
     - `db_backup/`（数据库旧备份）
     - `design/` 下的零散 `.png` 截图和 `.html` 演示（只留 `DESIGN_SYSTEM.md`）
     - `docs/screenshots/` 下的历史截图
     - `miniprogram/pages_backup_before_refactor/`（重构前的废弃页面代码）
   - *好处：让代码搜索变快，降低发版体积，让你看着不心烦。*

2. **冻结正式环境的日常开发用途**
   - 约定：正式环境（目前的 `cloud1-3g7709vbb44fe137`）只用于正式版运行
   - 不再在正式库里手工造测试数据
   - 不再直接拿正式环境调试新逻辑

3. **建立环境台账**
   - 记录当前有哪些云环境、云函数、数据库集合、环境变量
   - 记录正式环境中哪些集合属于核心用户资产

4. **做第一次正式备份**
   - 对正式数据库做完整导出
   - 给这次备份标注日期和用途，存放到安全的独立位置（不在代码库里）

5. **补一份发布禁令**
   - 没有备份，不发布
   - 没有测试环境验证，不发布
   - 涉及数据结构变更但没有回滚方案，不发布

#### 阶段 1 的交付物

- 干净清爽的项目目录结构
- 一份环境清单
- 一份正式数据库备份
- 一份发布前必须遵守的规则

##### 阶段 2：把开发与正式环境真正分开（跨端友好版）

#### 目标

让代码、云函数、数据库能明确区分“开发测试”和“正式生产”状态。**我们不使用物理分文件夹的土办法（那会毁了未来的跨端迁移），也不强绑定微信的原生环境判断接口，而是用最标准的“本地配置文件”来实现。**

#### 💡 图解原理：为什么改一行代码，环境就切了？

在“单代码库+配置文件”模式下，你的小程序代码里会多出一个叫 `env.js` 的文件。它的工作原理就像一个**火车轨道道岔**。

```text
[你的本地代码 (包含了所有页面、逻辑)]
        |
        | 读取 env.js 里的 CURRENT_ENV
        v
    【道岔：CURRENT_ENV = 'test' 还是 'prod'?】
        |
   +----+----+
   |         |
如果是'test' 如果是'prod'
   |         |
   v         v
[测试数据库] [正式数据库]
(随便折腾)  (真实用户数据)
```

当你把 `CURRENT_ENV` 写成 `'test'`，你在开发者工具里点任何按钮，存的数据都会飞到测试库。正式库里的数据安全得像放在保险箱里。

#### 💡 图解原理：代码是怎么“物理化”保存到正式环境的？

很多新手会困惑：“改一个配置怎么就变成正式版了？” 
这涉及到微信开发者工具的“上传”机制。代码**不是**在你点保存的那一刻就到了用户手机上的，它有一个“打包发货”的过程。

```text
[你的电脑]                       [微信服务器]                     [真实用户手机]

1. 你在本地把 env.js           
   改成 'prod'
        |
2. 点击工具右上角的 
   【上传】按钮  ===========> 3. 微信工具把你的代码压缩成
        |                       一个几十KB的 .wxapkg 物理文件包，
        |                       存到微信的云端服务器上。
        |                                |
        |                                v
        |                     4. 你登录微信公众平台后台，
        |                        把刚才传上去的包设为【体验版】
        |                        (此时测试人员可以扫码测试)
        |                                |
        |                                v
        |                     5. 测试没问题，你在后台点击【提交审核】
        |                        审核通过后点击【发布】
        |                                |
        |                                v
        |                     6. 这个包含了 'prod' 配置的物理包，
        |                        就被正式推给了所有用户。
```

**关键点在于：**
上传上去的那个包，是**固定**的、**静态**的。
因为你上传前，把代码里的字改成了 `'prod'`，所以打包进那个 `.wxapkg` 物理文件里的字，永远是 `'prod'`。真实用户下载了这个包，他们的手机运行起来，就只会连正式库。

而在你上传完之后，你可以立刻在自己电脑上把 `env.js` 改回 `'test'` 继续开发新功能。**你本地改回 test，绝不会影响微信服务器上已经打包好的 prod 版本。** 它们已经物理隔离了。

#### 这一阶段要完成什么

1. **云环境精简为两套（足够且省钱）**
   - **Test（测试环境）**：供你日常开发、跑联调，以及发体验版验收用。
   - **Prod（正式环境）**：即现有的 `cloud1-3g7709vbb44fe137`，仅供线上真实用户使用。
   - *结论：没必要建三套云环境，开发和体验版完全可以共用一套 Test 库，降低维护心智。*

2. **建立跨端友好的本地配置文件（单代码库隔离）**
   - **为什么不分三个文件夹？**
     因为未来如果要上 iOS/Android（比如用 uni-app/Taro/React Native），原生工程的配置非常复杂。如果分三个文件夹，你改一个 bug 要在三个文件夹里抄三遍代码，必定会漏改、错乱。
   - **正确做法：单代码库 + 配置切换**
     在代码里建一个简单的 `env.js` 配置文件：
     ```javascript
     // 开发时你手动把它改成 'test'，发正式版前你手动改成 'prod'
     export const CURRENT_ENV = 'test'; 
     ```
   - 修改 [app.js](file:///d:/第二曲线/识为鲜PinFresh/miniprogram/app.js)，让它读取这个 `CURRENT_ENV` 来决定连哪个云环境。
   - *好处：这才是业界标准的“土办法”。代码只有一份，不会精神分裂；未来跨端时，这种配置模式无缝兼容所有框架。*

3. **初始化测试环境的资源**
   - 在新建的 Test 云环境里，建好同名的数据库集合和上传同版的云函数。

4. **建立体验版验收通道**
   - 以后所有待上线版本，本地保持 `CURRENT_ENV = 'test'` 发体验版。
   - 先在 Test 库里做完整回归，确认没问题。
   - **发布前最后一步**：把 `CURRENT_ENV` 改成 `'prod'`，直接提审或发布正式版。

### 阶段 2 的交付物

- 三套环境分工说明
- 环境配置方案
- 开发版 / 体验版 / 正式版对应关系说明

### 这一阶段完成后，开发方式会怎么变化

变化很简单：

- 平时写代码：只连 dev
- 给自己或少量人验收：走 staging
- 面向正式用户：才用 prod

## 阶段 3：建立“数据安全修改”机制

### 目标

以后即使要改数据库结构、迁移用户数据、修云函数，也尽量做到可控。

### 这一阶段要完成什么

1. **定义哪些改动属于高风险**

以下都视为高风险改动：

- 改数据库结构
- 改 `_openid` 归属逻辑
- 改用户登录或身份判断
- 改云函数入参和出参
- 改购物清单、历史记录、食谱等核心集合

2. **建立迁移规则**

以后任何迁移脚本都必须满足：

- 先备份
- 可重复执行
- 失败后不直接删源数据
- 有执行日志
- 能统计迁移前后数量是否一致

3. **明确哪些数据不能直接删**

正式环境里这些数据默认不能手工删除：

- `users`
- `shopping_lists`
- `histories`
- `recipes`

除非满足：

- 已完成备份
- 已确认删除范围
- 已确认影响面
- 已写明回滚办法

4. **建立“先测再迁”的规则**

任何真实迁移前，都先在 staging 做一次完整演练。

### 阶段 3 的交付物

- 数据变更分级规则
- 数据迁移 SOP
- 发布前备份与回滚模板

### 这一阶段的意义

它解决的是：以后不是“能不能改”，而是“怎么改才不会把线上数据弄坏”。

---

## 阶段 4：建立长期可持续的迭代流程

### 目标

让你之后每次修复和迭代，都有一条稳定的、可重复的路线。

### 标准迭代流程

#### 1）接需求

先判断这次改动属于哪一类：

- **低风险**：文案、样式、小范围前端交互
- **中风险**：页面逻辑、云函数参数、列表数据写入
- **高风险**：登录、数据库结构、迁移、权限、正式环境配置

#### 2）在 dev 开发

- 先在开发环境实现
- 用测试账号和测试数据验证
- 不碰正式用户数据

#### 3）在 staging 验收

- 发体验版
- 按关键链路回归
- 检查历史兼容、老用户升级、核心路径是否正常

#### 4）准备发布

发布前固定检查：

- 是否已备份
- 是否有回滚方案
- 是否完成回归
- 是否涉及环境变量变更
- 是否涉及数据库变更

#### 5）正式发布

- 只发布已经在 staging 验过的内容
- 不在发布时临时顺手改其他问题
- 不把“顺便优化一下”混进正式发布

#### 6）发布后观察

- 观察关键云函数报错
- 观察用户核心链路
- 观察数据库是否出现异常数据增长、空字段、重复数据

## 5. 你作为小白，最需要先学会的 6 个概念

为了不被这些术语吓住，你先只要理解下面 6 个词：

### 1. 环境

就是“同一套程序的不同运行场地”。

- dev 是练习场
- staging 是彩排场
- prod 是正式营业场

### 2. 体验版

不是给正式用户用的，而是给内部验收和少量测试人看的版本。

### 3. 备份

就是出事后还能回来的“后悔药”。

### 4. 回滚

就是发现新版本有问题后，快速退回到上一版。

### 5. 迁移

就是把旧数据按新规则搬过去。

### 6. 回归测试

就是你改了一个地方以后，要再检查核心功能有没有被连带弄坏。

## 6. 我建议我们先做的优先级顺序

如果只按现实最重要的顺序来，我建议这样排：

### P0：必须先完成

这些不做，后面继续迭代风险很大：

1. 明确正式环境边界
2. 做正式库备份
3. 建立 dev / staging / prod 三套环境
4. 把小程序环境配置从“写死”改成“可控切换”
5. 建立正式发布前检查单

### P1：紧接着做

这些是为了保证以后每次迭代都稳：

1. 建立体验版验收流程
2. 建立数据库变更与迁移 SOP
3. 补全安全审核清单
4. 建立回滚流程

### P2：后续持续完善

这些不是第一天必须完成，但值得逐步补齐：

1. 操作日志和异常监控
2. 更细的权限控制
3. 更完整的数据巡检
4. 自动化回归或测试脚本

## 7. 这次筹备阶段我们不急着做的事

为了避免你一上来就陷入过度重构，我建议暂时不把以下内容作为第一批重点：

- 不先追求“代码绝对优雅”
- 不先追求“架构一步到位”
- 不先大面积重写页面
- 不先全面重做数据库

现在最重要的是：

**先把线上和开发隔开，先让发布变得可控。**

5. 发布流程
