# 识为鲜 PinFresh 正式上线后的安全迭代计划

> **制定初衷**：本计划的唯一目的，是让项目从“单人试验期”安全过渡到“上线产品期”。
> **核心最高纲领（已焊死）**：
> 1. **极度的安全感**：绝不能污染线上真实数据。任何修改必须有“手动强制锁死”的保底退路和“控制台一键回退”的后悔药。
> 2. **极度的防踩坑**：绝不为眼前的方便（如分物理文件夹）给未来跨端（iOS/Android）埋雷，所有隔离方案必须是跨端友好的。
> 3. **极度的直白可控**：拒绝黑盒魔法。改了哪行代码、微信是怎么物理化上传的、环境是怎么切换的，必须用大白话和图解明明白白地解释清楚。
>
> 只有在这三条底线都满足的前提下，我们才开始写下一行代码。

## 1. 先说结论

从现在开始，我们要把工作流切换成“带护栏的开发模式”。

最核心的变化只有 4 条：

1. **正式环境不再拿来日常开发和联调**
2. **真实用户数据不再拿来随便测试**
3. **任何会改数据结构、改云函数、改鉴权的动作，都先在测试环境验证**
4. **每次发布都要有备份、检查、回滚方案**

如果只记住一句话，那就是：

**以后默认在测试环境开发，在体验版验收，在正式环境做最小、可控、可回滚的发布。**

---

## 2. 为什么现在必须换开发方式

根据当前仓库状态，已经出现了几个典型风险：

- 小程序前端当前直接写死了单一云环境，说明开发、联调、线上很可能还在混用同一套云资源
- 项目里虽然已经有 `temp_` 集合与迁移逻辑，但它解决的是“游客转正式”的数据归属问题，不等于“开发环境和正式环境隔离”
- 安全审核清单还是未完成状态，说明上线后的运行治理还没有真正建立起来
- 一旦继续直接在正式库上改逻辑、跑测试、调云函数，就可能出现脏数据、误删、重复迁移、用户数据异常

也就是说，**你现在最需要补的，不是某一个功能，而是一整套“上线后继续开发”的护栏。**

---

## 3. 我们接下来采用的总策略

我建议采用下面这套策略：

### 策略 A：三套环境隔离

把云开发环境拆成三层：

- **开发环境 dev**：你自己日常写代码、跑联调、做实验
- **预发布环境 staging**：体验版验收、回归测试、模拟上线前状态
- **正式环境 prod**：只服务真实用户，不做随手测试

### 策略 B：三类数据分开

数据也分成三类看待：

- **开发测试数据**：随便造，允许清理
- **预发布验收数据**：接近真实，但不是正式用户资产
- **正式用户数据**：只允许通过正式发布逻辑产生，不能拿来做验证材料

### 策略 C：先建护栏，再做迭代

不建好护栏和兜底方案，绝不动核心业务代码：

1. **环境隔离与保底机制**（首要任务）
2. **大扫除与目录规范**（让代码清爽、搜索精准）
3. **数据变更分级与回滚SOP**（凡是动数据的，必须有后悔药）
4. **体验版验收强制通道**（拒绝本地测完直接发正式版）
5. **安全与发布检查单**（每次上线的最后一道防线）

### 策略 D：以后所有需求都走同一条路径

以后修一个 bug、加一个功能、改一个页面，都按这个顺序：

**需求确认 → 测试环境开发 → 测试环境验收 → 预发布回归 → 正式发布 → 发布后观察**

## 阶段 2：把开发与正式环境真正分开（跨端友好版）

#### 目标

让代码、云函数、数据库能明确区分“开发测试”和“正式生产”状态。**我们不使用物理分文件夹的土办法（那会毁了未来的跨端迁移），也不强绑定微信的原生环境判断接口，而是用最标准的“本地配置文件”来实现。**

#### 💡 图解原理：为什么改一行代码，环境就切了？

在“单代码库+配置文件”模式下，你的小程序代码里会多出一个叫 `env.js` 的文件。它的工作原理就像一个**火车轨道道岔**。

```text
[你的本地代码 (包含了所有页面、逻辑)]
        |
        | 读取 env.js 里的 CURRENT_ENV
        v
    【道岔：CURRENT_ENV = 'test' 还是 'prod'?】
        |
   +----+----+
   |         |
如果是'test' 如果是'prod'
   |         |
   v         v
[测试数据库] [正式数据库]
(随便折腾)  (真实用户数据)
```

当你把 `CURRENT_ENV` 写成 `'test'`，你在开发者工具里点任何按钮，存的数据都会飞到测试库。正式库里的数据安全得像放在保险箱里。

#### 💡 核心答疑：体验版转正式版，云环境怎么切？云函数怎么发？

很多小白在第一次接触多环境时，会有两个极其经典的灵魂拷问。这里我们用最直白的方式解释清楚。

**疑问 1：如果我发体验版时代码里写的是连 Test 库，那我点“转正式版”时，它怎么知道要切到 Prod 库？我是不是得重新改代码再上传一次？**

**答案是：不需要你手动改，微信帮你切。**

在我们的最终方案里，`env.js` 不是写死的一行字，而是利用了微信的一个神奇能力：**小程序能自己判断自己当前处于什么状态**。

它的真实代码逻辑是这样的：
```javascript
// env.js 伪代码
const version = 微信.获取当前版本状态(); // 它会返回 'develop'(开发版)、'trial'(体验版) 或 'release'(正式版)

if (version === 'release') {
  当前环境 = 'Prod正式库';
} else {
  当前环境 = 'Test测试库';
}
```

**工作流图解：**
1. 你在本地写完代码，点击【上传】。包裹里带着上面这段逻辑。
2. 你在后台把这个包裹设为【体验版】。
3. 你的手机扫码打开体验版，微信偷偷告诉代码：“喂，你现在是 `trial`（体验版）”。代码一算：“哦，那我要连 Test 库。”
4. 测试没问题，你在后台直接点【提交审核】->【发布】。**注意，你没有重新上传任何代码，包裹还是原来那个包裹。**
5. 真实用户打开正式版，微信偷偷告诉代码：“喂，你现在是 `release`（正式版）了”。代码一算：“卧槽，我要连 Prod 正式库！”

这就是**“一次打包，自动切库”**的魔法。你永远不需要在上线前一秒心惊胆战地去改代码。

**追问：万一微信这个自动判断失效了，或者我就是不放心，有什么退路吗？**
**有的，我们可以加一个“手动强制开关”作为保底。**

在 `env.js` 里，我们会留一行注释掉的代码：
```javascript
// 如果你害怕微信自动判断失效，就把下面这行取消注释，强行锁死环境
// export const FORCE_ENV = 'prod'; 
```
只要这行代码存在，代码就不管微信说什么，强行连正式库。这就相当于你手里握着一把**物理越权钥匙**，随时可以退回“手工切库”的土办法。

---

**疑问 2：云函数在开发中也要不断改，怎么保证测试库的云函数和正式库的云函数互不影响？**

**答案是：小程序前端代码是“一个包裹自动切”，但云函数是“两个平行宇宙，你指哪打哪”。**

云函数**不打包在小程序代码里**，它们是直接传到腾讯云服务器上的独立小脚本。

在微信开发者工具里，你的“云开发”有两个平行的宇宙：Test 宇宙 和 Prod 宇宙。
它们有各自独立的数据库，也有各自独立的云函数存放区。

**云函数的按需发布流：**
1. **日常开发**：你在工具右上角把云环境切换到 `Test`。然后你在 `cloudfunctions/analyze` 上点右键 -> 【上传并部署】。这个新写的云函数就飞到了 Test 宇宙。你随便怎么测，Prod 宇宙里的那个旧函数纹丝不动。
2. **体验版测试**：体验版连的是 Test 库，调用的自然也是 Test 宇宙里那个刚传上去的新函数。
3. **准备发正式版**：前端代码你在后台点发布就行了。但是云函数，你需要**手动**在工具右上角把环境切到 `Prod`，然后再次在 `analyze` 上点右键 -> 【上传并部署】。这时，经过验证的新函数才真正覆盖了正式环境的旧函数。

**追问：如果我手残，开发一半的半成品，不小心环境没切对，传到 Prod 覆盖了正式云函数怎么办？**
**不用慌，微信云开发控制台有“后悔药（版本回退）”。**

云函数每次上传，其实微信都偷偷帮你存了历史快照。
**补救步骤：**
1. 立即打开工具上的【云开发】面板。
2. 找到被你误覆盖的那个云函数，点击右侧的【版本与灰度】。
3. 你会看到之前的版本记录，直接把流量 **100% 切回上一个正常版本**。
4. 此时线上用户瞬间恢复正常。然后你再回本地，切回 Test 环境慢慢修你的代码。

**总结口诀：**
- **前端代码**：传一次，靠微信状态自动切库（自带手动开关保底）。
- **云函数代码**：写完先传 Test 测，测好切到 Prod 再传一遍（自带控制台一键回退保底）。

#### 这一阶段要完成什么

1. **云环境精简为两套（足够且省钱）**
   - **Test（测试环境）**：供你日常开发、跑联调，以及发体验版验收用。
   - **Prod（正式环境）**：即现有的 `cloud1-3g7709vbb44fe137`，仅供线上真实用户使用。
   - *结论：没必要建三套云环境，开发和体验版完全可以共用一套 Test 库，降低维护心智。*

2. **建立跨端友好的本地配置文件（单代码库隔离）**
   - **为什么不分三个文件夹？**
     因为未来如果要上 iOS/Android（比如用 uni-app/Taro/React Native），原生工程的配置非常复杂。如果分三个文件夹，你改一个 bug 要在三个文件夹里抄三遍代码，必定会漏改、错乱。
   - **正确做法：单代码库 + 配置切换**
     在代码里建一个简单的 `env.js` 配置文件：
     ```javascript
     // 开发时你手动把它改成 'test'，发正式版前你手动改成 'prod'
     export const CURRENT_ENV = 'test'; 
     ```
   - 修改 [app.js](file:///d:/第二曲线/识为鲜PinFresh/miniprogram/app.js)，让它读取这个 `CURRENT_ENV` 来决定连哪个云环境。
   - *好处：这才是业界标准的“土办法”。代码只有一份，不会精神分裂；未来跨端时，这种配置模式无缝兼容所有框架。*

3. **初始化测试环境的资源**
   - 在新建的 Test 云环境里，建好同名的数据库集合和上传同版的云函数。

4. **建立体验版验收通道**
   - 以后所有待上线版本，本地保持 `CURRENT_ENV = 'test'` 发体验版。
   - 先在 Test 库里做完整回归，确认没问题。
   - **发布前最后一步**：把 `CURRENT_ENV` 改成 `'prod'`，直接提审或发布正式版。- 确认没问题再准备正式提审或发布

#### 阶段 2 的交付物

- 两套环境（Test/Prod）分工说明
- `env.js` 环境配置方案
- 开发版 / 体验版 / 正式版对应关系说明

#### 🤝 我们的分工

- **你（主导者/业务确认）**：负责在微信开发者工具里点击“上传”，并在后台发体验版扫码验证；负责在正式发布前，告诉我“现在切 Prod”，以及自己动手切工具右上角的云环境发云函数。
- **我（AI 辅助/工程落地）**：负责写 `env.js` 代码；负责修改 `app.js` 里的动态判断逻辑；负责画图和给你讲明白这套代码是怎么跑的。

### 这一阶段完成后，开发方式会怎么变化

变化很简单：

- 平时写代码：只连 dev
- 给自己或少量人验收：走 staging
- 面向正式用户：才用 prod

## 阶段 3：建立“数据安全修改”机制

### 目标

以后即使要改数据库结构、迁移用户数据、修云函数，也尽量做到可控。

### 这一阶段要完成什么

1. **定义哪些改动属于高风险**

以下都视为高风险改动：

- 改数据库结构
- 改 `_openid` 归属逻辑
- 改用户登录或身份判断
- 改云函数入参和出参
- 改购物清单、历史记录、食谱等核心集合

2. **建立迁移规则**

以后任何迁移脚本都必须满足：

- 先备份
- 可重复执行
- 失败后不直接删源数据
- 有执行日志
- 能统计迁移前后数量是否一致

3. **明确哪些数据不能直接删**

正式环境里这些数据默认不能手工删除：

- `users`
- `shopping_lists`
- `histories`
- `recipes`

除非满足：

- 已完成备份
- 已确认删除范围
- 已确认影响面
- 已写明回滚办法

4. **建立“先测再迁”的规则**

任何真实迁移前，都先在 staging 做一次完整演练。

### 阶段 3 的交付物

- 数据变更分级规则
- 数据迁移 SOP
- 发布前备份与回滚模板

#### 🤝 我们的分工

- **你（主导者/业务确认）**：凡是动核心数据的改动，必须由你先下命令并确认“这批数据就算删了也没事”；迁移前你负责人工备份；出事后你负责在控制台一键回退。
- **我（AI 辅助/工程落地）**：负责写“幂等、可重复执行、有防错保护”的迁移脚本代码；绝不私自执行 `db.remove()`；负责给你列出每一步操作的后果说明。

## 阶段 4：建立长期可持续的迭代流程

### 目标

让你之后每次修复和迭代，都有一条稳定的、可重复的路线。

### 标准迭代流程

#### 1）接需求

先判断这次改动属于哪一类：

- **低风险**：文案、样式、小范围前端交互
- **中风险**：页面逻辑、云函数参数、列表数据写入
- **高风险**：登录、数据库结构、迁移、权限、正式环境配置

#### 2）在 dev 开发

- 先在开发环境实现
- 用测试账号和测试数据验证
- 不碰正式用户数据

#### 3）在 staging 验收

- 发体验版
- 按关键链路回归
- 检查历史兼容、老用户升级、核心路径是否正常

#### 4）准备发布

发布前固定检查：

- 是否已备份
- 是否有回滚方案
- 是否完成回归
- 是否涉及环境变量变更
- 是否涉及数据库变更

#### 5）正式发布

- 只发布已经在 staging 验过的内容
- 不在发布时临时顺手改其他问题
- 不把“顺便优化一下”混进正式发布

#### 6）发布后观察

- 观察关键云函数报错
- 观察用户核心链路
- 观察数据库是否出现异常数据增长、空字段、重复数据

#### 🤝 我们的分工（以后每次需求）

- **你（需求提出与最终拍板）**：提想法，点发布，看数据。发现问题后及时按“保底退路”撤回。
- **我（代码落地与风险把控）**：把你的一句话变成代码；写代码前先判断风险等级；在代码里埋好异常保护；每次交付都附带“验证清单”。

## 5. 你作为小白，最需要先学会的 6 个概念

为了不被这些术语吓住，你先只要理解下面 6 个词：

### 1. 环境

就是“同一套程序的不同运行场地”。

- dev 是练习场
- staging 是彩排场
- prod 是正式营业场

### 2. 体验版

不是给正式用户用的，而是给内部验收和少量测试人看的版本。

### 3. 备份

就是出事后还能回来的“后悔药”。

### 4. 回滚

就是发现新版本有问题后，快速退回到上一版。

### 5. 迁移

就是把旧数据按新规则搬过去。

### 6. 回归测试

就是你改了一个地方以后，要再检查核心功能有没有被连带弄坏。

## 6. 我建议我们先做的优先级顺序

如果只按现实最重要的顺序来，我建议这样排：

### P0：必须先完成

这些不做，后面继续迭代风险很大：

1. 明确正式环境边界
2. 做正式库备份
3. 建立 dev / staging / prod 三套环境
4. 把小程序环境配置从“写死”改成“可控切换”
5. 建立正式发布前检查单

### P1：紧接着做

这些是为了保证以后每次迭代都稳：

1. 建立体验版验收流程
2. 建立数据库变更与迁移 SOP
3. 补全安全审核清单
4. 建立回滚流程

### P2：后续持续完善

这些不是第一天必须完成，但值得逐步补齐：

1. 操作日志和异常监控
2. 更细的权限控制
3. 更完整的数据巡检
4. 自动化回归或测试脚本

## 7. 这次筹备阶段我们不急着做的事

为了避免你一上来就陷入过度重构，我建议暂时不把以下内容作为第一批重点：

- 不先追求“代码绝对优雅”
- 不先追求“架构一步到位”
- 不先大面积重写页面
- 不先全面重做数据库

现在最重要的是：

**先把线上和开发隔开，先让发布变得可控。**

---

## 8. 我建议的实际执行方式

等你确认这份计划后，我建议下一步按下面的输出方式继续推进：

### 第一步：出一份《环境与发布治理清单》

把需要准备的东西列清楚，例如：

- **第一件事：运行自动归档脚本，把废弃文件夹打包进 `_archive`**
- 三套环境分别怎么建
- 每套环境放什么
- 哪些配置要搬去环境变量
- 哪些人可以碰正式环境

### 第二步：出一份《实施任务拆解表》

把工作拆成可以逐项完成的任务：

- **大扫除类**：运行清理脚本，提交干净的 Git
- 配置类：修改 `app.js` 接入 `envVersion`
- 云函数类
- 数据类
- 发布类
- 验证类

### 第三步：按“先护栏，后功能”开工

优先做：

1. **大扫除与归档**
2. 环境隔离
3. 配置治理
4. 备份与回滚
5. 发布流程

然后再恢复常规功能迭代。

## 9. 本文档对应的项目判断

结合你当前项目，我的判断是：

- 现在最危险的问题不是某个单点 bug
- 而是“项目已进入正式运行阶段，但开发方式还停留在实验期”
- 因此，本轮最正确的动作不是立刻继续加功能
- 而是先补齐“正式上线后如何安全开发”的制度和基础设施
