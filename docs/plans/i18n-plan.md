# 知食角 (PinFresh) 国际化全局双语版本工作计划

## 项目背景

面向国际市场，推出"英文优先（默认英文），中文为辅"的双语版本小程序，让全球用户都能便捷使用知食角的食材识别、鲜度判断和菜谱推荐功能。

---

## 核心产品决策

| 决策项 | 选择 | 说明 |
|--------|------|------|
| 切换入口位置 | 「我的」页面 | 遵循“无感即最佳体验”，将低频的语言切换收纳至“我的”页面。提供 `🌐 语言 / Language` 入口，点击后弹出原生 ActionSheet 供用户选择 |
| 默认语言策略 | 跟随微信系统，非中文即英文 | 自动读取微信客户端语言，如果是 `zh_CN`、`zh_TW`、`zh_HK` 则默认中文；否则默认英文 |
| 历史数据处理 | 保持原有语言 | 已生成/收藏的采购单和识别结果保持生成时的语言，不实时翻译，降低成本并保证稳定性 |

## 阶段一：产品与架构奠基（Infrastructure）

### 1.1 更新 PRD
将"全局英文版本"正式纳入 MVP 的 P0 范围，确立"英文为主，中文为辅"的产品基调。

### 1.2 构建 i18n 基础字典
- 在 `miniprogram/utils/` 下建立双语字典文件：
  - `en.js`（主干，英文为主）
  - `zh.js`（辅助，中文）
- 提取全局所有硬编码文本（如"拍照识别"、"正在分析食材特征"等）

### 1.3 全局语言状态管理与探针
- 修改 `app.js`，在小程序冷启动时调用 `wx.getSystemInfoSync().language` 探测微信客户端语言
- **核心逻辑**：
  - 系统语言为 `zh_CN`, `zh_TW`, `zh_HK` → 默认中文
  - 其他所有语言（包括 `en`, `ja`, `ko` 等）→ 默认英文
- 将用户手动切换的结果存入 `wx.setStorageSync('pf_lang')`，优先级高于系统默认
- 编写全局翻译函数 `t(key)`，供所有页面调用

## 阶段二：前端界面双语化与排版优化（UI Localization）

### 2.1 全页面文本替换
遍历以下页面，将硬编码中文替换为动态绑定变量：
- `pages/index`（首页）
- `pages/result`（结果页）
- `pages/list`（采购单）
- `pages/my`（我的）

**WXML 替换示例**：
```html
<!-- 原 -->
<text class="title">拍照识别</text>
<!-- 改 -->
<text class="title">{{i18n.camera_title}}</text>
```

**JS 替换示例**：
```javascript
// 原
wx.showToast({ title: '正在分析食材特征...' })
// 改
wx.showToast({ title: t('analyzing') })
```

### 2.2 排版防崩溃机制（重点）
- **问题**：英文单词长度通常比中文长 1.5 倍左右，容易导致 UI 溢出
- **解决方案**：
  - 增加 CSS `word-break: break-word` 防止长单词撑破容器
  - 调整 `flex` 布局的换行机制
  - 重点检查：按钮、卡片、食材名称、菜谱名称等区域的宽度适配
  - 适当增加英文版本的 `padding` 和 `line-height` 以提升可读性

## 阶段三：AI 云端分析双语适配（Backend AI Adaptation）

### 3.1 打通前后端语言链路
- 小程序端调用 `analyze` 云函数时，传入当前语言标识：
  ```javascript
  wx.cloud.callFunction({
    name: 'analyze',
    data: { imageBase64, lang: 'en' } // 或 'zh'
  })
  ```

### 3.2 动态调整大模型 Prompt
修改 `cloudfunctions/analyze/index.js`：

**英文模式 Prompt 示例**：
```
You are a professional chef and food quality expert. Analyze the provided ingredient image and return a pure JSON object with these exact fields:
- "ingredient_name": (string) Ingredient name in English
- "ingredient_desc": (string) A brief objective description
- "taste": (string) Flavor profile (e.g., "fresh and sweet", "rich")
- "texture": (string) Texture (e.g., "firm", "tender")
- "similar": (string) 1-3 similar ingredients as common nouns only
- "freshness_level": (string) "Fresh" | "Average" | "Not Fresh"
- "freshness_reason": (string) Visual assessment reason
- "recipes": (array) 2-3 recipes, each with "recipe_name" and "ingredients_needed" (string array)

Return ONLY JSON, no markdown, no additional text.
```

**中文模式 Prompt**：使用现有的中文 Prompt 逻辑。

### 3.3 返回结构一致性
确保无论中英文模式，AI 返回的 JSON 结构完全一致，这样前端解析代码无需改动。

- 用户点击切换语言后，更新全局状态 `app.globalData.language`

- 优势：节省翻译 API 成本，保证历史数据稳定性

## 共用 vs 分开的划分原则

### ✅ 应该共用的部分（Single Source of Truth）

| 类别 | 共用原因 | 示例 |
|------|----------|------|
| **架构基础设施** | 与语言无关 | 云函数调用封装 `utils/analyze.js`、认证模块 `utils/auth.js`、小程序初始化配置 |
| **业务逻辑** | 跨语言完全一致 | 页面跳转逻辑、图片选择逻辑、存储读写逻辑、数据解析逻辑 |
| **UI 结构与样式** | 布局一致，只是文本长度不同 | 所有 `.wxss` 中的布局、Flex 盒子、组件结构 |
| **非文本资源** | 语义化图标全球通用 | `images/icons/` 下所有 SVG 图标、背景图 `market-bg.jpg` |

### ⚡️ 应该分开的部分（Localized）

| 类别 | 分开原因 | 示例 |
|------|----------|------|
| **硬编码文本** | 必须翻译 | 导航栏标题 "知食角"、按钮文案 "拍照识别"、提示语 "正在分析..."、错误消息 |
| **动态翻译内容** | 由 AI 根据语言返回不同版本 | 食材名称、菜谱名、鲜度描述、佐料列表（这部分是"数据层"的分层） |

## 三层分离架构（保证未来维护便利）

```
┌─────────────────────────────────────────────────────────┐
│                    展示层 (UI Layer)                    │
│   所有页面 WXML + JS + WXSS (结构/样式/交互 完全共用)    │
│   ┌─────────────────────────────────────────────────┐  │
│   │  {{ i18n.camera_title }}  ← 动态读取翻译变量      │  │
│   └─────────────────────────────────────────────────┘  │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                  翻译服务层 (i18n Layer)                │
│   ┌──────────────────┐    ┌──────────────────┐         │
│   │   en.js (英文)   │    │   zh.js (中文)   │         │
│   │   主干字典       │    │   辅助字典       │         │
│   └──────────────────┘    └──────────────────┘         │
└─────────────────────────────────────────────────────────┘
                           ↓
┌─────────────────────────────────────────────────────────┐
│                  基础设施层 (Foundation)                │
│   utils/analyze.js | auth.js | app.js (逻辑完全共用)    │
└─────────────────────────────────────────────────────────┘
```

### 🔑 关键设计点

| 设计点 | 实现方式 | 维护优势 |
|--------|----------|----------|
| **字典驱动** | 所有 UI 文本从 `i18n.xxx` 变量读取，变量名一一对应字典 Key | 新增文案只需改字典文件，不动 UI 代码 |
| **数据结构不变** | AI 返回的中英文 JSON 结构完全一致（字段名相同，只是值用不同语言） | 前端解析代码只需写一套，兼容两种语言 |
| **CSS 弹性适配** | 预设 `word-break: break-word`，避免英文溢出 | 样式只需调一次，长文本自动适配 |
| **语言状态集中管理** | 在 `app.js` 中统一定义 `t()` 函数和语言切换逻辑 | 切换语言时全局生效，无需逐页处理 |

## 维护 Checklist（未来迭代参考）

每次迭代或修复时，可以按这个顺序检查：

1. **文案改动？** → 只改 `en.js` / `zh.js` 字典文件
2. **样式/布局改动？** → 修改 `.wxss`，中英文同步生效

5. **AI 输出内容改动？** → 修改云函数 Prompt，同时兼容两种语言

## 兼容性设计策略（降低风险）

1. **新鲜度映射表**：在 `app.js` 或独立 `utils/freshness.js` 中维护中英文映射表
   ```javascript
   const freshnessMap = {
     zh: { '新鲜': 'fresh', '一般': 'normal', '不太新鲜': 'bad' },
     en: { 'Fresh': 'fresh', 'Average': 'normal', 'Not Fresh': 'bad' }
   };
   ```

2. **佐料连接符**：统一使用 `i18n.list_separator`（中文 `"、"`，英文 `", "`）

3. **分享文案模板**：
   - 中文：`"今晚准备做【${name}】，快来看看需要买什么！"`
   - 英文：`"Tonight we're making 【${name}】! Here's what you need to buy:"`

4. **默认兜底值**：云函数返回的默认值也需要支持多语言

5. **Mock 数据分离**：`utils/analyze.js` 中的 `MODE` 判断后，根据 `app.globalData.language` 加载对应的 Mock 文件

## 回归测试重点

完成国际化改造后，**必须验证以下场景**：

1. ✅ 切换到英文后，所有静态 UI 文本是否正确显示为英文
2. ✅ 切换到中文后，所有静态 UI 文本是否正确显示为中文
3. ✅ 英文模式下，AI 返回的食材名称、菜谱名、佐料是否为英文
4. ✅ 中文模式下，AI 返回的食材名称、菜谱名、佐料是否为中文
5. ✅ 分享给好友的卡片标题和描述是否根据当前语言正确生成
6. ✅ 历史记录（中文环境生成）切换到英文后，是否保持原语言（不翻译）
7. ✅ 英文模式下，长文本是否正确换行，不溢出容器
