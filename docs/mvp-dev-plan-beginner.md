# 《知食公社》微信小程序 MVP 开发计划（小白跟练版）

这份文档的目标：你不需要懂太多技术背景，只要按步骤做，就能把 PRD 里定义的 MVP 做到“可验收、可演示”。

对应 PRD：`docs/mvp-prd.md`  
对应设计：`design/herbal_harmony/DESIGN.md` 与 `design/_2/_3/_4`

---

## 0. 你将做出什么（MVP 三页闭环）

你最终会拥有 3 个页面（已存在，但需要补齐能力）：

- 首页：拍照/上传 → 开始识别
- 结果页：食材名 + 鲜度结论 + 2-3 个做法 → 生成采购单
- 采购单页：佐料清单勾选 + 一键分享 + 再拍一次 + 清除本地数据

核心闭环（必须跑通）：

1. 打开小程序 → 自动建立登录态（静默）拿到 `userId`
2. 拍照/上传 → 分析中 → 识别结果
3. 选做法 → 生成采购单 → 勾选 → 分享 → 再拍一次

### 1.4 GitHub 备份与恢复（必须先做，保证“随时可回退”）

目标是做到两件事：

- 任何电脑上都能从 GitHub 拉下代码并跑起来（备份/恢复）
- 出问题可以快速回滚到上一个“能演示”的版本（可恢复）

你要做什么（按顺序）：

1. 把项目推到 GitHub（或企业 Git 平台），并确认远端是你有权限的仓库
2. 分支约定（MVP 最简版）：
   - `main`：任何时候都保持可运行、可演示

3. 提交约定（MVP 最简版）：

## M1：把“用户会话”做出来（静默登录）

### 目标

冷启动就有稳定的 `userId`，不弹授权弹窗。并把下面两个键写入本地存储：

- `pf_user_id`: string
- `pf_auth_source`: `"cloud_openid" | "wx_login" | "anonymous"`

### 你要做什么

1. 在小程序端新增一个统一的登录初始化函数 `initAuth()`
2. 在 `App.onLaunch()` 中调用 `initAuth()`
3. 云开发可用时，优先通过云函数返回 openid
4. 如果拿不到 openid，就生成匿名 id 并持久化

建议实现文件位置（建议新增）：

- `miniprogram/utils/auth.js`

需要新增云函数：

- `cloudfunctions/login`：返回 openid

### 验收（PRD 用例 00）

- 首次打开小程序：不弹授权弹窗
- 任意页面 `wx.getStorageSync("pf_user_id")` 有值
- `pf_auth_source` 是三者之一

## M2：做“清除本地数据/退出登录”

### 目标

用户点击后清掉本地缓存，并能立即继续使用拍照识别，不会卡死。

### 你要做什么

1. 统一本地存储键（建议全部以 `pf_` 开头）
2. 实现 `logout()`：
   - 清空 `pf_user_id`、`pf_auth_source`
   - 清空业务缓存（例如最近一次识别、上次图片、已选做法）
   - 然后再调用一次 `initAuth()` 重新建会话
3. 放一个入口（MVP 推荐放在“采购单页”底部）

建议实现文件位置：

- `miniprogram/utils/auth.js`（与 M1 放一起）

### 验收（PRD 用例 06）

- 先跑一遍识别流程
- 点击“清除本地数据”
- 再次拍照识别还能继续用，且 `pf_user_id` 会重新生成/获取

## M3：把“分析模块”做稳（mock/云函数统一入口 + 错误提示）

### 目标

你能清晰区分：

- 成功：正常展示结果
- 失败：Toast 提示“识别失败/网络错误/请重试”，并允许重试

同时保证返回结构稳定，至少包含：

- `ingredient_name`
- `ingredient_desc`
- `freshness_level`
- `freshness_reason`
- `recipes`（2-3 个，且每个都有 `recipe_name`、`ingredients_needed`）

### 你要做什么

1. 把分析请求封装成单一入口（项目里已有 `miniprogram/utils/analyze.js`，继续完善即可）
2. 增加超时保护（例如 10-15 秒）
3. 对返回结构做兜底补齐（缺字段就填默认值）
4. 失败时不要直接静默回 mock（否则你没法验收 PRD 的失败用例）

### 模型接通（必须显式落地在 M3 里）

你要明确“最终是谁在跑模型”，并把链路固定下来，避免后期反复返工。MVP 推荐默认方案：

- 小程序 → 云函数 `analyze` → 模型服务（外部 API 或你自建服务）

你要做什么（最小可用）：

1. 在 `miniprogram/utils/analyze.js` 固定一个“单一入口”，内部可切换：
   - `mock`：只用于离线演示/无网络
   - `cloudfunction`：用于真实联调与验收
2. 定义并固化“接口契约”（入参/出参/错误码）：
   - 入参：图片文件 ID 或临时 URL + `userId` + 请求来源
   - 出参：必须满足本节上面列出的字段结构
   - 错误：区分 `network_error` / `timeout` / `model_error` / `bad_response`
3. 云函数侧实现与保护（最小可用）：
   - 超时与重试策略（至少有超时）
   - 把模型 Key 放到云函数环境变量，不写进仓库
   - 把模型返回映射成稳定结构，缺字段兜底

### 验收（PRD 用例 05 + 01/02）

- 正常网络：拍照 → 结果页能展示完整信息
- 模拟失败：能看到 Toast，并可再次点击重试

## M4：结果页体验补齐（默认选中 + 佐料摘要 + 异常态）

### 目标

结果页信息完整、可选做法、能生成采购单；当结果不完整时，有明确兜底页面。

### 你要做什么

1. 做法默认选中第 1 个（当前已有）
2. 做法卡片补充“佐料摘要”（显示前 3 个佐料 + “等”）
3. 增加异常态：
   - 没有 `recipes` 或结构缺失时：提示“结果不完整，建议换角度再拍”
   - 提供“再拍一次 / 返回首页”

### 验收（PRD 用例 03 + 异常态）

- 选择做法 2 → 生成采购单页展示对应佐料
- 让 `recipes` 为空时：能进入异常态，不崩溃

## M5：采购单页补齐（勾选 + 分享文案 + 再拍一次 + 清除数据入口 + 收藏功能）

### 目标

采购单页能勾选、能分享、能快速回到首页再拍，并能清除本地数据。同时支持收藏佐料清单到云端。

### 你要做什么

1. 勾选逻辑升级为 `checkedMap`（建议）或继续用 index 数组（可先不改）
2. 分享卡片标题建议为：`食材名 + 做法名 + 佐料清单`
3. 分享路径 MVP 先回到首页（避免分享打开后找不到采购单）
4. 增加“清除本地数据/退出登录”按钮
5. 增加“收藏”按钮：
   - 点击后将当前佐料清单（食材名+做法名+已勾选佐料）写入云数据库的 `favorites` 集合
   - 收藏成功后 Toast 提示"已收藏"
   - 可选择留在当前页面或返回首页

### 验收（PRD 用例 04 + 06 + 07）

- 取消勾选几项 → 点击分享 → 出现微信分享面板
- 点击清除数据后能继续使用
- 点击收藏 → Toast提示成功 → 数据写入云端
