# 性能优化与现有资产保全专项规划

> **核心基调**：本项目已处于正式上线运行期。所有的性能优化必须让步于**“极度安全感”**与**“输入输出守恒（I/O Consistency）”**。宁可牺牲一点性能，也绝不能引发老用户白屏、数据丢失或业务流程断裂。

基于 2026-04-16 的 MiniTest 报告（10 项待改进），本计划将性能优化与现有功能/数据的**保全护栏**深度结合，确保每一次改动都有退路。

---

## 🛡️ 总体保全法则（最高优执行）

在执行任何一类性能优化前，必须强制执行以下保全动作：
1. **测试环境隔离**：所有优化代码仅在 `develop` 分支进行，并在微信开发者工具连 `Test` 库跑通。
2. **I/O 守恒评估**：改动前，列出该模块原有的所有输入参数和输出结构。优化后，底层数据结构（尤其是存入云数据库和 Storage 的字段）**必须 100% 保持原样**。
3. **老缓存兜底**：永远假设线上用户带着旧版 Storage 缓存。改动存储读取逻辑时，必须兼容旧状态。
4. **不碰核心业务流**：本次优化尽量不动“大模型多路并行解析”、“V2数据库嵌套结构”和“食谱拆分解耦”这三大核心引擎的底层逻辑。

---

## 一、 内存泄漏治理（最安全，建议首选）
**【待改进项】**：页面销毁时未及时移除 `wx.on` 类型监听器。
**【优化方案】**：在各页面的 `onUnload` 中成对添加 `wx.off...`。
**【功能保全护栏】**：
* **影响面确认**：排查全局的 `EventBus`、`wx.onThemeChange`、`wx.onNetworkStatusChange` 等。
* **保全措施**：绝不修改监听器内部的业务逻辑，仅仅是补充销毁动作。确保像“页面降维回退依赖 `onShow` 静默刷新”这种多入口状态同步逻辑不被误杀。

## 二、 setData 通信瘦身（高收益，需谨慎保全）
**【待改进项】**：setData 传入未绑定在 WXML 的变量；页面跳转响应慢。
**【优化方案】**：剥离不需要在 WXML 渲染的巨大 JSON 字段，只将视图层需要的字段放入 `setData`。
**【数据保全护栏（输入输出守恒）】**：
* **致命风险**：如果盲目删减 `setData` 里的字段，可能会导致用户点击“保存”或“加入清单”时，由于 `this.data` 里找不到那些隐藏字段，导致存入数据库的数据残缺！
* **保全措施**：**数据双轨制**。不需要渲染的庞大字段（如大模型原始结果），禁止放入 `setData`，但**必须**原封不动地挂载到 `this.xxx`（页面实例属性）上。当需要发往云函数或存库时，将 `this.data` 和 `this.xxx` 拼合，**确保最终发往后端的 Payload 结构与老版本 100% 一致**。

## 三、 启动包袱与 Storage 治理（涉及生命周期，需防白屏）
**【待改进项】**：App.onLaunch 耗时；启动阶段重复获取 storage；首屏前调同步 API。
**【优化方案】**：将 `app.js` 中反复读取的 Storage 建立内存映射（挂在 `globalData`）；推迟非关键初始化。
**【逻辑保全护栏（老用户兼容）】**：
* **致命风险**：改动启动逻辑极易导致鉴权时序错乱，引发老用户误入 onboarding 页面，或拿不到 OpenID。
* **保全措施**：**死守 `authReadyPromise` 锁**。任何对 `app.js` 的瘦身，都绝对不能破坏现有的 `initAuth` 鉴权时序。`utils/auth.js` 自动恢复 `has_onboarded` 和 `userInfo` 缓存的逻辑必须完整保留。只改变“读取的方式（从反复读硬盘改为读内存）”，绝不改变“读取的时机和结果”。

## 四、 视图渲染释放（涉及长列表）
**【待改进项】**：过大的 WXML 节点数目。
**【优化方案】**：历史记录/清单长列表的分页或按需渲染。
**【数据保全护栏】**：
* **致命风险**：重构列表渲染可能会破坏已有的事件绑定（如传参错误）。
* **保全措施**：**参数守恒**。由于咱们经历了 V2 数据库规整（从 `ing._id` 变成了 `ing.id`），在优化 WXML 节点时，必须严格保留 `data-id="{{item.id}}"` 等传参逻辑。此外，针对“正在进行中的清单禁止重复加入自身”的防御逻辑，必须在优化后的列表中原样生效。

## 🚀 实施排期建议（带安全阀）

1. **Step 1: 摘取安全果实（无损级）**
   * 修复 `wx.on` 监听器未解绑。
   * 全局搜索多余的同步 API（`xxxSync`），评估替换风险。
   * **验证**：不涉及任何数据变更，直接在 Test 体验版点按验收。
2. **Step 2: 核心链路无感瘦身（保全级）**
   * 针对识物结果页、食谱详情页，实施 `setData` 双轨制瘦身。
   * **验证**：修改后，必须走通“识物 -> 纠偏编辑 -> 加入清单 -> 查看食谱 -> 收藏食谱”全链路，并对比数据库中写入的 JSON 结构，确认与优化前**一模一样（I/O 守恒）**。
3. **Step 3: 启动项治理（最高危）**
   * 优化 `app.js` 和 Storage 读取。
   * **验证**：必须用带有旧缓存的老用户账号、无缓存的新账号分别在 Test 环境冷启动，确保不会出现白屏或路由错乱。
