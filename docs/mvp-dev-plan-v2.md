# 《知食角》微信小程序 MVP V2 开发计划（个人中心与历史记录）

在完成了 V1 的核心闭环后，我们现在要给应用加入**用户体系（头像/昵称）**和**历史记录（多食材回溯）**。这不仅能解决用户在菜市场买多个菜时“忘了之前识别过啥”的痛点，也是我们未来做分享裂变和商业化的基础。

对应 PRD：`docs/mvp-prd.md` 第 8 节  
设计指引：沿用 `design/herbal_harmony` 的视觉规范。

---

## 0. 你将做出什么（V2 目标）

我们将把小程序从“用完即走的一次性工具”升级为“能记住你的采购小助手”。你最终会拥有：

1. **底部的全局导航（TabBar）**：包含“首页”和“我的”两个 Tab。
2. **“我的”页面**：可以点击设置头像和昵称，这会真正同步到云端数据库。
3. **历史记录列表**：在“我的”页面展示你拍过的食材和做法，点击可以直接回溯查看。

核心闭环（V2 必须跑通）：
- 打开小程序 → 进入“我的”页面 → 设置头像/昵称（显式授权）
- 首页拍照识别 → 成功生成采购单 → 自动将这条记录存入云端
- 返回“我的”页面 → 看到历史记录 → 点击某条历史 → 重新进入对应的采购单

## 1. 开发前准备

在开始写代码前，我们要准备好**云开发数据库（Cloud DB）**。

1. 打开微信开发者工具的 **云开发控制台**。
2. 点击 **数据库**，创建两个集合（Collection）：
   - `users`：用来存用户的基本信息（头像、昵称、注册时间）。
   - `histories`：用来存每次识别的历史记录。
   - `favorites`：用来存用户收藏的佐料清单（MVP M5 功能）。
3. 把这两个集合的权限设置为：**所有用户可读，仅创建者可读写**（这样用户只能看到/修改自己的数据，非常安全且不用写复杂的云函数鉴权）。

## M8：全局导航（TabBar）与“我的”页面骨架

### 目标
把小程序的单页面结构升级为带底部导航（TabBar）的结构，并把“我的”页面建出来。

### 你要做什么
1. 在 `miniprogram/app.json` 中配置 `tabBar`，包含两个页面：
   - 首页：`pages/index/index`
   - 我的：`pages/my/index` (需要新建这个文件夹和页面文件)
2. 在新建的 `pages/my/index.wxml` 中写一个简单的骨架：
   - 顶部区域：默认头像 + “点击登录”文案。
   - 列表区域：“我的历史记录”占位。
3. 之前放在“采购单页”底部的“清除本地数据”入口，现在可以挪到“我的”页面里了。

### 验收
- 编译后，小程序底部出现了 TabBar。
- 点击“我的”可以顺利切换页面，能看到头像占位和清除数据按钮。

## M9：实现微信原生授权登录（头像与昵称）

### 目标
用微信最新的官方能力，让用户安全、合规地设置头像和昵称，并存入云数据库的 `users` 集合。

### 你要做什么
1. 了解微信新规：现在不能用 `getUserProfile` 一键获取了，必须让用户自己选。
2. 在 `pages/my/index` 页面：
   - 使用 `<button open-type="chooseAvatar">` 来获取用户选择的头像。
   - 使用 `<input type="nickname">` 来获取用户填写的昵称。
3. 获取到头像和昵称后，将头像图片上传到**云存储**（因为微信返回的是临时路径）。
4. 拿到云存储的头像 URL 和昵称后，写入或更新到云数据库的 `users` 集合中（匹配当前的 `userId` / `openid`）。

### 验收
- 进入“我的”页面，点击头像能唤起微信的头像选择框。
- 点击昵称能唤起微信的昵称填写键盘。
- 保存后，打开云开发控制台 -> 数据库 -> `users` 集合，能看到多了一条带有你头像 URL 和昵称的记录。

## M10：识别结果自动写入历史记录

### 目标
用户每次成功识别并进入“采购单页”时，我们在后台悄悄把这次记录存进数据库。

### 你要做什么
1. 在“结果页”点击“生成采购单”的逻辑中（或者分析成功的统一回调里），增加写入数据库的代码。
2. 向 `histories` 集合插入一条记录，包含：
   - `ingredient_name`：食材名
   - `selected_recipe`：用户选中的做法（包含做法名和佐料）
   - `createdAt`：当前时间（服务器时间）
3. 确保这个写入动作是**异步**的，不要阻塞用户进入采购单页面的速度。

### 验收
- 走一遍正常的拍照识别流程，选个做法，生成采购单。
- 打开云开发控制台 -> 数据库 -> `histories` 集合，确认新增了一条记录，且内容准确。

## M11：“我的”页面展示历史记录与回溯

### 目标
让用户能在“我的”页面看到过去拍过的食材，并能点进去重新查看。

### 你要做什么
1. 在 `pages/my/index.js` 的 `onShow` 生命周期里，去云数据库查询 `histories` 集合（按 `createdAt` 倒序排列，限制拉取前 20 条）。
2. 在页面上把列表渲染出来：
   - 左侧：食材名称（比如“鲈鱼”）
   - 右侧：做法名称（比如“清蒸鲈鱼”） + 时间
3. 给每条记录绑定点击事件：点击后，带着该条记录的 `selected_recipe` 和 `ingredient_name` 跳转到 `pages/list/index`（采购单页）。

### 验收
- 杀掉小程序重新进，进入“我的”页面，能看到刚才测试生成的历史记录。
- 点击某一条记录，直接跳转到了采购单页，并且佐料清单完全对应。

## M11-1：“我的”页面展示收藏列表

### 目标
让用户能在“我的”页面看到自己收藏的佐料清单，并能点进去重新查看。

### 你要做什么
1. 在“历史记录”列表旁边或下方，增设“我的收藏”区域
2. 查询云数据库的 `favorites` 集合（按 `createdAt` 倒序排列，限制拉取前 20 条）
3. 在页面上把收藏列表渲染出来：
   - 左侧：食材名称 + 做法名称
   - 右侧：收藏时间
4. 给每条收藏记录绑定点击事件：点击后，带着该条记录的佐料清单跳转到 `pages/list/index`（采购单页）

### 验收
- 进入“我的”页面，能看到收藏的佐料清单列表
- 点击某一条收藏记录，直接跳转到了采购单页，并且佐料清单对应

## M12：UI 统一与边界情况打磨（联调与发布）

### 目标
把 V2 的体验做到“丝滑”，处理好各种没网、没登录的边界情况。

### 你要做什么
1. **未登录状态保护**：如果用户处于“匿名状态”（比如没开启云开发或拿不到 openid），“我的”页面提示“请开启云服务以保存历史”，屏蔽数据库读写，防止报错。
2. **样式打磨**：“我的”页面的列表 UI 参照设计稿规范（圆角、卡片、字体层级）进行优化。
3. **空状态**：如果用户没有历史记录，展示一个友好的缺省图或文案（如：“去拍个食材试试看吧～”）。

### 验收
- 全链路跑通：换个新微信号（或清空缓存）体验，从授权头像 -> 拍照 -> 存历史 -> 回溯历史，全程无阻断、无报错。
- 断网情况下进入“我的”页面不崩溃，有合理的提示。
